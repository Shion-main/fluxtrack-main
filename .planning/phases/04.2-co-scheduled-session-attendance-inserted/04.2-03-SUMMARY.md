# Phase 04.2 Plan 03: Checker Online Seam Summary

**Wired the Checker online seam (`web/checker.py:_apply_action`) to the Plan-01 merge core per modality: an online Verify now flips its online co-scheduled siblings present (ACTIVE + MERGED + shared actual_start + audited, NO CheckerValidation) and an online Flag-not-present fails the whole online group ABSENT — each inside the SAME `transaction.atomic()` as the anchor write, via the D-01 course_code / V-room key with `teams_link` never consulted — while the F2F flag path is left exactly record-only, proven by 6 green tests and a clean 228-test regression (only the 2 documented pre-existing login-static errors).**

## Accomplishments
- **Task 1 — online Verify propagation:** the VERIFIED branch of `_apply_action(online=True)` now wraps the anchor `ACTIVE + ONLINE_MANUAL` save and `propagate_merged_present(session, session.actual_start, actor=request.user)` in one `transaction.atomic()`. A single online Verify on `ONL200 V01` flips sibling `ONL200 V02` to `ACTIVE + checkin_method=MERGED` with the anchor's shared `actual_start` and one `session.merged_present` AuditLog. The sibling gets **no** `CheckerValidation` (D-09) so `verified_by_checker` stays False; the anchor keeps its real `ONLINE_MANUAL` method and its own validation. The merge fires via shared `course_code` with `teams_link` blanked on both rows — proving no `teams_link` merge key (Post-Research Clarification #1).
- **Task 2 — online Flag-ABSENT propagation + F2F record-only:** the FLAG_NOT_PRESENT branch wraps the anchor `ABSENT` save and `propagate_merged_absent(session, actor=request.user)` in one `transaction.atomic()`, so one online not-present fails the whole online group ABSENT (each sibling audited `session.merged_absent`). The helper's SCHEDULED-only status guard leaves an already-ACTIVE sibling untouched. The **F2F** path in `_apply_action` was left byte-for-byte record-only: an F2F Flag-not-present writes the `CheckerValidation` + notifies but does **not** change the merged F2F group's status (still SCHEDULED), leaving the none-held case to the JOB-02 sweep (D-08).

## Decisions Made
- **Same D-01 key online, no teams_link clause.** Online siblings are found by the shared `course_code` / V-room arm exactly like F2F; a test blanks `teams_link` on both online rows and asserts the merge still fires and equals the pure detector's set — there is provably no `teams_link` dependency (Post-Research Clarification #1).
- **Per-modality not-present (D-07).** Online Flag-not-present propagates ABSENT immediately (matching today's online-writes-ABSENT); the F2F flag path stays record-only and defers to the sweep (D-08). The F2F branch of `_apply_action` was not touched.
- **Honest CHK-04 coverage (D-09).** Merge-filled siblings get no `CheckerValidation`; only the genuinely-scanned anchor is checker-verified. Asserted with a negative `verified_by_checker is False` on the sibling and a positive on the anchor.
- **Atomicity over the anchor.** The `propagate_*` call sits inside the anchor-write `transaction.atomic()` so a rollback of the anchor rolls back the whole merged group. The upstream online re-gate (`_online_action`: ownership + on-duty + actionable + idempotency) is unchanged and still runs before any write.

## Threat Mitigations Applied
- **T-04.2-05 (Tampering — online fan-out):** merge-fill runs ONLY after the existing `_online_action` re-gate (ownership via `online_checker`, active online-duty, actionable) and the atomic `cache.add` idempotency pass; the merged set is server-derived via the D-01 faculty-scoped key, so a forged/replayed POST cannot fan ABSENT/ACTIVE to a foreign group. - **T-04.2-04b (Tampering — CHK-04 integrity):** siblings get NO `CheckerValidation` (D-09), asserted by `test_merge_filled_sibling_has_no_checker_validation`.
