# Phase 04.2: Co-Scheduled Session Attendance - Context

**Gathered:** 2026-07-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Attendance-layer fix so that **merged sections** — one instructor teaching two or
more section-rows *together as a single class* at the same time — are handled by a
single presence event. One faculty check-in (or one Checker verification) covers
every row in the merged group, and the JOB-02 sweep never falsely marks a merged
sibling Absent. Attendance behavior only: **no roster/data merge**, sections stay
distinct in the schema and in reporting.

**In scope:** detecting a merged group from existing session fields; propagating
present/absent across the group at check-in, checker-verify, and checker-not-present;
recording merge-filled rows traceably.

**Out of scope:** any change to rosters/section identity; staggered/overlapping-time
grouping (we lock exact-same-start); a persisted "merge group" table; reporting UI
(Phase 6); notifications surface (Phase 5).
</domain>

<decisions>
## Implementation Decisions

### Merge Definition
- **D-01:** A **merged class** = 2+ `Session` rows with the **same faculty**, the
  **same `scheduled_start`** (exact, same date), **AND (same `room` OR same
  `schedule.course_code`)**. Different-room *and* different-course ⇒ NOT a merge.
  Rationale: catches the GARAY case via same course (`MMA116` across the A408-A /
  A408-B divisible-room halves) and catches two different courses sharing one
  literal room.
- **D-02 (terminology):** The project term is **"merged sections/sessions,"** not
  "co-scheduled." It is one physical class taught to 2+ sections together, stored as
  separate rows only to preserve the distinct rosters. Use "merged" in code/docs.
- **D-03:** Merge groups are detected **dynamically** from existing fields at
  check-in / verify / sweep time. No new "merge group" model, no migration for
  grouping, no roster change.

### Presence Propagation — Mechanism (A)
- **D-04:** Any presence event that sets one session present marks **ALL** rows of
  its merged group present (`status = ACTIVE`, shared `actual_start`) in the **same
  DB transaction**. Because siblings become `ACTIVE`, the sweep skips them with **no
  sweep-side change** on the present path. Directly satisfies ROADMAP success
  criteria #1 and #2.
- **D-05:** Idempotent — only `SCHEDULED → ACTIVE` (mirror the existing
  `web/scan.py` `_apply` and `jobs.py` sweep guards). A row already `ACTIVE` or
  `ABSENT` is left as-is.
- **D-06:** Triggers = **both** faculty self check-in (`web/scan.py`) **and** a
  Checker's Verify (`web/checker.py`), including the **online** case: one Checker
  verification via the shared MS Teams link covers the sibling online (V-room)
  sessions.

### Absence Propagation
- **D-07:** A Checker's **"not present"** flag fails **ALL** merged rows (mark the
  group Absent) — it is one class, so one absence covers every row.
- **D-08:** The no-event case needs **no special sweep logic**: merged rows share
  `scheduled_start`, so the existing `sweep_no_shows` marks them all Absent together
  via the shared `is_no_show_past_grace` predicate.

### Record & Integrity
- **D-09:** The row that was actually scanned/verified keeps its **real
  `checkin_method`** (QR / manual / `ONLINE_MANUAL`). Merge-filled rows are stamped
  with a **distinct `checkin_method = MERGED`** (new `CheckinMethod` choice), the
  **same `actual_start`**, plus an **`AuditLog`** entry (e.g.
  `event_type="session.merged_present"`, payload `{"merged_from": <scanned session
  pk>}`). Reporting still counts them held; HR/audit can distinguish a direct scan
  from a merge-fill.

### Claude's Discretion (planner decides)
- Where the **pure merge-sibling detector** lives (e.g. `scheduling/resolver.py` or a
  new `scheduling/merge.py`) and its unit tests — follow the existing
  `is_no_show_past_grace` "pure + coupling-tested" pattern.
- Exact transaction ordering and reuse of the `SCHEDULED→` idempotency guard.
- Whether absence propagation (D-07) writes one `AuditLog` per row or one for the group.
</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `is_no_show_past_grace` (`scheduling/resolver.py`): the single no-show predicate.
  **Do NOT fork it.** The fix works precisely by making merged siblings `ACTIVE`, so
  this predicate never fires on a satisfied merge — the coupling guarantee is preserved.
- `CheckinMethod` enum (`scheduling/models.py`): add a `MERGED` choice for D-09.
- `AuditLog` write pattern (`scheduling/jobs.py` `session.marked_absent`): mirror its
  shape for the `session.merged_present` audit.

### Established Patterns
- **Pure, coupling-tested decision functions** (`is_no_show_past_grace`) — the new
  merge-sibling detector should be pure and unit-tested the same way.
- **MSSQL "materialize candidates before mutating"** (`jobs.py` uses `list(...)`
  before `save()` to avoid HY010 / single-active-result-set) — any batch mutation
  over merged rows must do the same.
- **Only-`SCHEDULED`→transition idempotency** (`web/scan.py` `_apply`, `jobs.py`
  sweep) — reuse for present-propagation.

### Integration Points
- `web/scan.py:79-82` (faculty check-in `_apply`) and `:116-120` (force-handover):
  after the scanned session goes `ACTIVE`, propagate present across the merged group.
- `web/checker.py:282-285` (online Verify) and the verify/flag action path
  (~`:642-682`): propagate **present** on Verify across the group; propagate
  **Absent** on Flag-not-present across the group (D-07).
- `scheduling/jobs.py` `sweep_no_shows`: **no change** for the present path (`ACTIVE`
  rows are already skipped); confirm merged rows cross grace together (shared
  `scheduled_start`).
- Merge key fields: `Session.faculty`, `Session.scheduled_start`, `Session.room`, and
  `Session.schedule.course_code` (course/section live on the related `Schedule`).

</code_context>

<specifics>
## Specific Ideas

- **Canonical test fixture — the GARAY case:** course `MMA116`, sections A301 (46
  enrolled, room A408-A) and A302 (24 enrolled, room A408-B), M/W 3:45 slot, same
  faculty `cdgaray`. One check-in on either row must mark both present; a
  not-present on either must fail both. Non-merged sessions must be unaffected.
- **Scale to keep in mind:** ~129 instructor+day+time slots hold 2+ concurrent
  sections across 54 of 200 instructors (~27%) — physical (divisible-room halves)
  and online (shared Teams link across sibling V-rooms).

</specifics>

<deferred>
## Deferred Ideas

- **Accepted tradeoff (not a defect):** the merge key uses *exact same start time*
  and *(same room OR same course)*. Staggered/overlapping-time merges and
  mixed-modality-within-one-merge are NOT handled; if such data surfaces in the term,
  revisit. A single instructor booked into two genuinely-separate rooms with
  different courses at once is treated as NOT a merge (it cannot be one class) and is
  a scheduling conflict, out of scope here.
- Superseded during discussion: an earlier tentative "any same-time pair = merged"
  scope was **replaced** by the precise D-01 merge key.

None — discussion stayed within phase scope.

</deferred>
