## Summary

This is a **brownfield attendance-layer fix**, not a greenfield build. Every integration point named in CONTEXT.md was read and verified against the current source this session — signatures, line ranges, and the surrounding idempotency/MSSQL patterns all still match. No external packages are introduced; the entire change is a new pure detector + call-sites in three existing seams (`web/scan.py`, `web/checker.py`, `scheduling/jobs.py` is *no-change*), one new `CheckinMethod.MERGED` enum choice, and its metadata migration.

The mechanism (D-04) is precise and low-risk *on the present path*: any presence event that drives one session `SCHEDULED → ACTIVE` also flips its merged siblings `SCHEDULED → ACTIVE` with a **shared `actual_start`** inside **one `transaction.atomic()`**, so the existing sweep (which only touches `SCHEDULED`) naturally skips them — no sweep-side change, directly satisfying success criteria #1 and #2. The single highest-risk area is MSSQL cursor semantics: the propagation must **materialize the sibling set before mutating** (the same `list(...)`-before-`.save()` guard `jobs.py` already documents for HY010 / single-active-result-set), and it should prefer a **single filtered `.update()`** over a mutate-while-iterate loop.

Two landmines need the planner's (and likely the user's) attention before they become bugs: (1) the **online shared-Teams-link merge** may not be caught by the locked D-01 key if sibling V-rooms differ in *both* room and course_code — D-01 keys on `room OR course_code`, never `teams_link`; and (2) **D-07 (checker "not present" fails the whole group)** collides with the existing F2F convention that a checker flag is *record-only* and leaves status to the sweep. Both are surfaced below with recommendations; neither re-litigates a locked decision.

**Primary recommendation:** Add a pure `scheduling/merge.py` with a coupling-tested `merged_sibling_ids(anchor, candidates)` predicate (D-01 truth table), then call it from the `CHECKED_IN` and `ROOM_OCCUPIED` branches of `web/scan.py:_apply` and the online-Verify / Flag-not-present branches of `web/checker.py:_apply_action`, each wrapping anchor-write + group propagation in one `transaction.atomic()` and using a filtered `.update()` for the state change plus a bulk AuditLog write.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Merge Definition**
- **D-01:** A **merged class** = 2+ `Session` rows with the **same faculty**, the **same `scheduled_start`** (exact, same date), **AND (same `room` OR same `schedule.course_code`)**. Different-room *and* different-course ⇒ NOT a merge. Rationale: catches the GARAY case via same course (`MMA116` across the A408-A / A408-B divisible-room halves) and catches two different courses sharing one literal room.
- **D-02 (terminology):** The project term is **"merged sections/sessions,"** not "co-scheduled." It is one physical class taught to 2+ sections together, stored as separate rows only to preserve the distinct rosters. Use "merged" in code/docs.
- **D-03:** Merge groups are detected **dynamically** from existing fields at check-in / verify / sweep time. No new "merge group" model, no migration for grouping, no roster change.

**Presence Propagation — Mechanism (A)**
- **D-04:** Any presence event that sets one session present marks **ALL** rows of its merged group present (`status = ACTIVE`, shared `actual_start`) in the **same DB transaction**. Because siblings become `ACTIVE`, the sweep skips them with **no sweep-side change** on the present path. Directly satisfies ROADMAP success criteria #1 and #2.
- **D-05:** Idempotent — only `SCHEDULED → ACTIVE` (mirror the existing `web/scan.py` `_apply` and `jobs.py` sweep guards). A row already `ACTIVE` or `ABSENT` is left as-is.
- **D-06:** Triggers = **both** faculty self check-in (`web/scan.py`) **and** a Checker's Verify (`web/checker.py`), including the **online** case: one Checker verification via the shared MS Teams link covers the sibling online (V-room) sessions.

**Absence Propagation**
- **D-07:** A Checker's **"not present"** flag fails **ALL** merged rows (mark the group Absent) — it is one class, so one absence covers every row.
- **D-08:** The no-event case needs **no special sweep logic**: merged rows share `scheduled_start`, so the existing `sweep_no_shows` marks them all Absent together via the shared `is_no_show_past_grace` predicate.

**Record & Integrity**
- **D-09:** The row that was actually scanned/verified keeps its **real `checkin_method`** (QR / manual / `ONLINE_MANUAL`). Merge-filled rows are stamped with a **distinct `checkin_method = MERGED`** (new `CheckinMethod` choice), the **same `actual_start`**, plus an **`AuditLog`** entry (e.g. `event_type="session.merged_present"`, payload `{"merged_from": <scanned session pk>}`). Reporting still counts them held; HR/audit can distinguish a direct scan from a merge-fill.

- Exact transaction ordering and reuse of the `SCHEDULED→` idempotency guard.
- Whether absence propagation (D-07) writes one `AuditLog` per row or one for the group.

### Deferred Ideas (OUT OF SCOPE)
- **Accepted tradeoff (not a defect):** the merge key uses *exact same start time* and *(same room OR same course)*. Staggered/overlapping-time merges and mixed-modality-within-one-merge are NOT handled; if such data surfaces in the term, revisit. A single instructor booked into two genuinely-separate rooms with different courses at once is treated as NOT a merge (it cannot be one class) and is a scheduling conflict, out of scope here.

- Out of scope (from Phase Boundary): any change to rosters/section identity; a persisted "merge group" table; reporting UI (Phase 6); notifications surface (Phase 5).

### Pattern 1: Pure detector + apply-layer propagation (mirror `is_no_show_past_grace`)
**What:** A pure function decides *which rows merge*; the web apply layer does the ORM writes. No queries inside the pure function.
**When to use:** For the D-01 detector — it must be unit-testable with plain objects, no DB.
**Example:**
```python
# scheduling/merge.py  (NEW — pure, no ORM, no timezone.now())
# Source: mirrors scheduling/resolver.py:39-52 "pure + coupling-tested" convention
def merged_sibling_ids(anchor, candidates):
    """Return the set of candidate ids that merge with `anchor` under D-01.

    A candidate merges when it is a DIFFERENT row with:
      - same faculty_id, AND
      - same scheduled_start (exact aware instant), AND
      - (same room_id OR same course_code).

    Pure: `anchor` and `candidates` are Session-like objects exposing
    .id/.faculty_id/.scheduled_start/.room_id/.course_code. The caller
    materializes them (select_related('schedule')) and supplies course_code.
    No ORM, no now() — coupling-tested against the live call-site the same way
    the sweep re-affirms is_no_show_past_grace.
    """
    out = set()
    for c in candidates:
        if c.id == anchor.id:
            continue
        if c.faculty_id != anchor.faculty_id:
            continue
        if c.scheduled_start != anchor.scheduled_start:
            continue
        if c.room_id == anchor.room_id or c.course_code == anchor.course_code:
            out.add(c.id)
    return out
```

### Pattern 2: Atomic group state change via filtered `.update()` (MSSQL-safe)
**What:** One `transaction.atomic()`; state change via a single filtered `UPDATE`; audit via `bulk_create`. Avoids the mutate-while-iterate cursor trap entirely.
**When to use:** For the propagation write in both seams.
**Example:**
```python
# inside web/scan.py:_apply, CHECKED_IN branch (illustrative)
# Source: transaction + materialize-before-mutate pattern from scheduling/jobs.py:56-73
from django.db import transaction
from scheduling.merge import merged_sibling_ids

with transaction.atomic():
    session.status = SessionStatus.ACTIVE
    session.actual_start = now
    session.checkin_method = method          # anchor keeps REAL method (D-09)
    session.save(update_fields=["status", "actual_start", "checkin_method"])

    # candidates: this faculty's same-start rows today, course_code carried in.
    candidates = list(
        Session.objects.filter(faculty_id=session.faculty_id,
                               scheduled_start=session.scheduled_start)
        .exclude(pk=session.pk)
        .select_related("schedule"))
    for c in candidates:                     # attach course_code for the pure fn
        c.course_code = c.schedule.course_code
    session.course_code = session.schedule.course_code
    sib_ids = merged_sibling_ids(session, candidates)

    # Idempotent group flip (only SCHEDULED transitions), single UPDATE — no
    # open SELECT cursor held across the write, so no HY010.
    fill_ids = list(Session.objects.filter(pk__in=sib_ids,
                                           status=SessionStatus.SCHEDULED)
                    .values_list("pk", flat=True))
    Session.objects.filter(pk__in=fill_ids).update(
        status=SessionStatus.ACTIVE, actual_start=now,
        checkin_method=CheckinMethod.MERGED)
    AuditLog.objects.bulk_create([
        AuditLog(actor=user, event_type="session.merged_present",
                 target_type="session", target_id=str(pk),
                 payload={"merged_from": session.pk}) for pk in fill_ids])
```

### Anti-Patterns to Avoid
- **Forking `is_no_show_past_grace`** or adding a sweep-side "sibling exemption" (that was ROADMAP option B, NOT chosen). D-04/D-08 make the sweep untouched by construction; adding sweep logic breaks the single-predicate coupling guarantee.
- **Mutating rows while a SELECT cursor is still open** (`.iterator()` / looping a live queryset and `.save()` inside) → MSSQL HY010 "Function sequence error" (MARS off). Materialize first, or use one `.update()`.
- **Comparing `scheduled_start` by `.date()` or by time-of-day only.** D-01 is *exact same aware instant*. Filter/compare the full `DateTimeField`.
- **A second round trip that re-reads siblings after the anchor save without the same-transaction wrap** — violates D-04 ("same DB transaction") and opens a window where the sweep could interleave.
- **Creating `CheckerValidation` rows for merge-filled siblings.** D-09 asks only for `checkin_method=MERGED` + `AuditLog`. Fabricating validations would inflate CHK-04 coverage metrics falsely (see Open Q3).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Atomic multi-row state flip | Manual per-row `save()` loop inside an open cursor | `transaction.atomic()` + one filtered `.update()` | Idempotent, single statement, sidesteps HY010 entirely |
| "Only transition SCHEDULED" idempotency | New guard logic | Reuse the `status=SCHEDULED` filter idiom from `scan.py:_apply` (ABSENT branch) and `jobs.py:65-68` | Proven pattern; keeps all three paths consistent |
| No-event group Absent | New sweep branch / sibling exemption | Existing `sweep_no_shows` unchanged (D-08) | Shared `scheduled_start` → all cross grace together via one predicate |
| Bulk audit writes | Loop of `AuditLog.objects.create()` (N cursors) | `AuditLog.objects.bulk_create([...])` after materializing ids | One INSERT batch, cursor-safe |

**Key insight:** The design's elegance is that almost all of it is *subtraction of risk* — the sweep, the grace predicate, and the reporting layer need **zero** changes. The only genuinely new correctness surface is the atomic present-propagation write, and Django's `transaction.atomic()` + filtered `.update()` cover it with battle-tested primitives.

## Common Pitfalls

### Pitfall 1: MSSQL HY010 on mutate-while-iterate
**What goes wrong:** Looping a live `Session` queryset and calling `.save()`/`AuditLog.create()` inside raises `HY010 Function sequence error (SQLFetch)` — pyodbc allows one active result set (MARS off).
**Why it happens:** The SELECT cursor is still open when the INSERT/UPDATE fires.
**How to avoid:** Materialize with `list(...)` before mutating (as `jobs.py:56` does), or perform the state change as a single filtered `.update()` (recommended). There is already a regression test to mirror: `test_batch_of_no_shows_all_marked_absent` (`scheduling/tests.py:348-360`).
**Warning signs:** Passes on SQLite dev DB, fails only on LocalDB/MSSQL — write the batch (N≥2 siblings) test on a `TestCase` that hits the configured DB.

### Pitfall 2: `scheduled_start` exactness across timezone round-trip
**What goes wrong:** A merge is missed because two "same-time" starts compare unequal.
**Why it happens:** Naive vs aware, or microsecond drift.
**How to avoid:** Both siblings are materialized by the same path (`_aware(date, time)` in materialize) so they are identical aware instants; Python `==` on aware datetimes compares instants. The round-trip is already proven safe (`DatetimeRoundTripTests`, `scheduling/tests.py:204-222`). Filter with the full `scheduled_start=...`, never `.date()`.
**Warning signs:** GARAY fixture passes but a cross-midnight-UTC slot fails — add a Manila-instant merge case.

### Pitfall 3: Online sibling not caught by D-01 (shared Teams link, distinct room + course)
**What goes wrong:** Success criterion #3 (one online verify covers online siblings) silently fails for V-room groups that differ in *both* `room` and `course_code`.
**Why it happens:** D-01 keys on `room OR course_code` — **never `teams_link`**. If sibling online sections share only the Teams link, they are not a merge under D-01.
**How to avoid:** Verify against real term data whether online sibling V-rooms share `course_code` (they usually do — same course, multiple sections). If any group shares *only* the Teams link, D-01 as locked will not cover it. **This is a locked-decision feasibility risk, not a re-litigation** — flag to the user (see Open Q1 / Assumptions A1).
**Warning signs:** Physical GARAY case green, but an online group still gets a sweep-Absent sibling.

### Pitfall 4: D-07 vs the F2F "record-only flag" convention
**What goes wrong:** Marking F2F merged siblings ABSENT on a checker flag would diverge from the existing single-session F2F behavior, which is deliberately **record-only** (`_apply_action` F2F branch writes the flag + notifies but leaves status to the sweep — see `web/checker.py:264-266`, "no silent status override, research Open Q2"). Only the *online* branch authoritatively sets ABSENT.
**Why it happens:** D-07 says a checker "not present" fails ALL merged rows; the online path already writes ABSENT, but the F2F path does not.
**How to avoid:** Scope D-07 propagation to match each modality's existing single-session semantics — **online** flag propagates ABSENT to online siblings immediately; **F2F** flag stays record-only and relies on D-08 (the sweep marks the whole SCHEDULED group Absent together). Confirm with the user before coding (see Open Q2).
**Warning signs:** An F2F merged group where the anchor is left SCHEDULED (record-only) but siblings are force-ABSENT — an inconsistent group state.

### Pitfall 5: Merge-fill overwriting a checked-out or absent sibling
**What goes wrong:** A late check-in resurrects a sibling the sweep already marked ABSENT, or clobbers a sibling that checked out early.
**Why it happens:** Forgetting the `status=SCHEDULED` filter on the `.update()`.
**How to avoid:** D-05 — the filtered `.update(... )` **must** carry `status=SCHEDULED`. ACTIVE/ABSENT/COMPLETED siblings are left untouched. (Because merged rows share `scheduled_start`, in practice they cross grace together, so this is an edge, but keep the guard.)

### Sampling / Edge Strategy
- **Property/truth-table sampling (the coupling-tested core):** The D-01 detector is a pure predicate over four independent axes — `faculty` (same/diff), `scheduled_start` (same/1-min-off), `room` (same/diff), `course_code` (same/diff). Exhaustively enumerate the meaningful combinations (a small Cartesian table) in `SimpleTestCase` — this is the Nyquist "sample every state-change boundary" requirement for the decision function. Critical rows: same-course/diff-room = MERGE (GARAY); same-room/diff-course = MERGE; diff-room AND diff-course = NOT merge; diff-faculty = NOT; 1-minute start offset = NOT (exactness boundary).
- **Coupling test:** Assert the live call-site (scan `_apply`) marks exactly the sibling set the pure `merged_sibling_ids` returns for the same fixture — the same "resolver ⇔ predicate" coupling `CouplingIntegrityTests` (`scheduling/tests.py:146-160`) enforces for the sweep. This prevents the seam and the pure fn from silently diverging.
- **Held-out / real-shape fixture:** Build the **GARAY canonical fixture** (course `MMA116`, sections A301/room A408-A and A302/room A408-B, same faculty `cdgaray`, same M/W 3:45 start) plus a **non-merged control** (different faculty or different start) and an **online sibling pair** sharing `course_code`. One check-in on either GARAY row must flip both; a genuinely-missed control must still go ABSENT.
- **MSSQL edge (must run on LocalDB, not SQLite):** the N≥2 batch propagation test proves no HY010 — mirror `test_batch_of_no_shows_all_marked_absent`.

### Applicable ASVS Categories
| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | No new auth surface; existing `@login_required` / `@checker_required` unchanged. |

| V6 Cryptography | no | None. |

### Known Threat Patterns for Django/MSSQL attendance-write
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| One scan marking a *different* instructor's sessions present | Elevation of Privilege | D-01 `same faculty` filter — merge set is scoped to `faculty_id=anchor.faculty_id`; assert in a security test. |
| Forged/replayed checker POST fanning out ABSENT to a group | Tampering | Reuse existing server-side re-gate + idempotency (`cache.add`) in `checker.py`; merge-fill runs only after the anchor passes the existing gate. |
| Partial write leaving a group half-flipped | (integrity) | `transaction.atomic()` wraps anchor + propagation (D-04). |
| ORM injection via crafted ids | Tampering | Parameterized ORM (`filter(pk__in=...)`); no raw SQL. |
