<objective>
Build the shared data foundation every other Phase 5 plan consumes: the per-user
mute-preference model, the `Notification.pushed_at` outbox stamp, and the single
category->type map with its filter helpers. This is the one source of truth (D-06)
that both the in-app list filter (05-04) and the push filter (05-03) import, so the
two can never disagree — mirroring the `notify()` single-write-path discipline.

Purpose: Guarantee mute semantics are defined exactly once. Encode the three
owner-approved resolutions: (1) unmapped types stay always-shown/never-mutable,
(2) `weekly_report_ready` is defined now as the Phase 5<->6 contract, (3) the
category groups are exactly Room / Reports / System per D-04.
Output: `NotificationMute` model, `pushed_at` migration, `ops/notifications.py`
map+helpers, and a full unit-test file (Nyquist Wave 0 gap closure).
</objective>

  <action>
In `ops/models.py`, add a `NotificationMute` model implementing per-user mute-by-category-group
(D-04/D-05): fields `user` (ForeignKey to settings.AUTH_USER_MODEL, on_delete=CASCADE,
related_name="notification_mutes"), `category` (CharField max_length=20 holding a
NotificationCategory value), `created_at` (auto_now_add). Add `class Meta` with
`unique_together = [("user", "category")]`. The PRESENCE of a row means that category is muted
for that user; ABSENCE means unmuted, so default is everything-unmuted (D-05) with no seed rows.

(the outbox stamp the 05-03 scheduler job sets after a push pass; NULL means not yet processed),
and add to its `Meta` an index `models.Index(fields=["user", "read_at"])` supporting the unread-badge
query. Keep the existing `ordering = ["-created_at"]`.

Keep the module docstring's SRS reference style. Then generate the migration with
`py -3.12 manage.py makemigrations ops` (expected name 0004_notificationmute_pushed_at). Do NOT
hand-edit migration internals beyond what makemigrations produces.
