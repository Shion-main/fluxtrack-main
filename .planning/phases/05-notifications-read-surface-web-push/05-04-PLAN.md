<objective>
Build the in-app read surface (NOTIF-01): a global context processor supplying poll_ms + unread
count + the VAPID public key to every page, a polled bell partial (badge + dropdown preview), a
full-page /notifications history, auto-mark-read on OPEN only, and a mute settings surface — all
filtered through the single map from 05-01 so mute suppresses the list exactly as it suppresses push.

Purpose: Make the events notify() already writes visible to every role via the established htmx poll
idiom (D-02), with read semantics per D-03 (auto-read on view, keep-all) and mute per D-05. The
read/write surface stays audit-silent by design.
Output: context processor + registration, the notification views + routes, the templates, and the
NOTIF-01/NOTIF-03 tests.

DELIBERATE EXCEPTION (do not "fix"): the mark-read and mute-toggle writes emit NO AuditLog. The
notification surface is audit-silent to match notify()'s own no-audit rule (CONTEXT + RESEARCH
Anti-Patterns). Flagged here so the plan-checker and executor treat the absence as intended.
</objective>

<task type="auto">
  <name>Task 2: Notification views (bell/dropdown/list + mute settings) + routes + tests</name>
  <files>web/notifications.py, web/urls.py, web/tests_notifications.py</files>
  <action>
Create `web/notifications.py` (one module per surface, Convention). All views `@login_required` and
strictly scoped to `request.user` (V2/V4 access control; never expose another user's rows — threat
T-05-09). Import visible_qs, unread_count, _mark_read, muted_types, NotificationCategory, CATEGORY_TYPES
from ops.notifications.

- `bell(request)`: READ-ONLY poll endpoint. Render `notifications/_bell_inner.html` with the unread
  count and a small preview (latest few visible_qs rows). It MUST NOT mark anything read — the badge
  only clears on open (D-02; RESEARCH Anti-Pattern: never mark read on the poll).
- `dropdown(request)`: render `notifications/_rows.html` for the latest visible_qs rows, THEN call
  `_mark_read` on those shown rows (D-03 auto-read on open). No AuditLog (deliberate).
- `list_page(request)`: full-page `notifications/list.html` over visible_qs (cap to a reasonable
  recent slice), THEN `_mark_read` the shown rows (D-03). No AuditLog.
- `settings_page(request)`: render `notifications/settings.html` with the three NotificationCategory
  groups and whether each is currently muted for request.user (NOTIF-03 mute UI).
- `mute_toggle(request)`: `@require_http_methods(["POST"])`. Toggle a NotificationMute row for
  (request.user, category) — create if absent, delete if present (presence = muted, D-05). Validate
  the posted category is a valid NotificationCategory value; ignore/400 otherwise. Return the updated
  settings partial or a redirect. Audit-silent (deliberate exception, consistent with the surface).

Wire routes in `web/urls.py` importing the new module: `notifications/bell` -> bell,
`notifications/dropdown` -> dropdown, `notifications` -> list_page, `notifications/settings` ->
settings_page, `notifications/mute` -> mute_toggle. Use names notif_bell/notif_dropdown/
notifications/notif_settings/notif_mute.

Create `web/tests_notifications.py` (Django TestCase). Cover: (a) badge = unread visible count and
excludes muted (D-02/D-05); (b) GET bell does NOT change read_at for any row (poll never marks read);
(c) GET dropdown and GET /notifications set read_at on shown rows and drop the badge to 0 (D-03);
(d) mark-read writes NO AuditLog — assert AuditLog.objects.count() is unchanged across a dropdown/list
open; (e) muting ROOM via POST /notifications/mute removes room_event rows from the list while an
unmapped checker_flag stays visible (owner #1); (f) all endpoints require login and never return
another user's notifications. ASCII-only messages.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> notification views | Authenticated users request their own notifications + toggle their own mutes |
| notification text -> HTML | Stored title/body rendered into the dropdown/list |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-09 | Information Disclosure | bell/dropdown/list/settings views | high | mitigate | Every view `@login_required` and queries only `request.user` rows (visible_qs(request.user)); no PK-addressed cross-user access (V4). |
| T-05-11 | Tampering | POST /notifications/mute | medium | mitigate | `@require_http_methods(["POST"])` + Django CSRF (global hx-headers X-CSRFToken); category validated against NotificationCategory; acts only on request.user. |
| T-05-10 | Tampering | notification title/body render | medium | mitigate | Django template auto-escaping; never `|safe` on notification text (XSS via a crafted room/course string). |
| T-05-12 | Repudiation | audit-silent mark-read/mute | low | accept | Intentional per notify() discipline; the read surface does not audit. Domain actions that spawned the notifications already carry their own AuditLog. |
</threat_model>
