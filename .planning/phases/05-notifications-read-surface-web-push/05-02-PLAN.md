  <action>
In `requirements.txt`, add a new heading comment `# Web push (NOTIF-02) — VAPID send; pulls
py-vapid + http-ece, uses existing cryptography` and the pin `pywebpush>=2.3,<3`. Install it with
`py -3.12 -m pip install "pywebpush>=2.3,<3"` (pure-Python; cryptography wheel already present, no
Windows build gotcha).

In `config/settings.py`, add a `# --- Web push (VAPID, NOTIF-02) ---` block AFTER the existing env
helpers, reading three values via the existing `env()` helper: `VAPID_PUBLIC_KEY = env("VAPID_PUBLIC_KEY", "")`,
`VAPID_PRIVATE_KEY_PATH = env("VAPID_PRIVATE_KEY_PATH", "")`, `VAPID_SUB = env("VAPID_SUB", "")`.
Empty defaults keep the app booting when push is unconfigured (the 05-03 sender treats an empty key
path as "push disabled"). Add `"push_outbox_interval_seconds": 15` to the `FLUXTRACK_POLICY` dict
with an inline comment tying it to D-09 (scheduler-driven push cadence; policy-driven, never a magic
number per Convention rule #3). Do NOT register any context processor here — that belongs to 05-04.

In `.gitignore`, add `*.pem` and a `keys/` directory entry so the VAPID private key can never be
committed (threat T-05-03). `.env` is already ignored.

In `.env.example`, add commented placeholders for VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY_PATH
(e.g. `keys/private_key.pem`), and VAPID_SUB (e.g. `mailto:ifo@mmcm.edu.ph`) with a note that real
values are generated in Task 3 and the PEM is never committed. If `.env.example` does not exist,
create it with only these placeholders (do not dump other secrets).

<task type="auto">
  <name>Task 3: Generate the VAPID keypair and populate .env (gitignored)</name>
  <files>.env.example</files>
  <action>
Generate the keypair using the py-vapid CLI that ships with pywebpush: run `vapid --gen` inside a
new gitignored `keys/` directory to produce `keys/private_key.pem` (and public_key.pem), then run
`vapid --applicationServerKey` to obtain the base64url application server key. Append (do not
overwrite) to the local `.env` file: `VAPID_PRIVATE_KEY_PATH=keys/private_key.pem`, `VAPID_PUBLIC_KEY=`
<the printed application server key>, and `VAPID_SUB=mailto:ifo@mmcm.edu.ph` (placeholder mailto the
developer can change to the real admin address). Print an ASCII-only confirmation (no emoji /
box-drawing, Windows cp1252 per Convention rule #4). Confirm `git check-ignore keys/private_key.pem`
reports the path as ignored. The private key is a secret (threat T-05-03): never echo its contents,
never `git add` the PEM. This file is the only artifact touched in git terms (.env.example note), but
the runtime effect is the populated .env + generated PEM.

If `vapid` is not on PATH, invoke it as `py -3.12 -m py_vapid` (the module entry point) with the same
subcommands.
  </action>
  <verify>
    <automated>py -3.12 -c "import os; from pathlib import Path; from dotenv import load_dotenv; load_dotenv('.env'); p=os.environ.get('VAPID_PRIVATE_KEY_PATH',''); k=os.environ.get('VAPID_PUBLIC_KEY',''); assert k and p and Path(p).exists(), 'VAPID env not populated'; print('vapid ok')"</automated>
  </verify>
  <done>keys/private_key.pem exists and is gitignored; .env has non-empty VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY_PATH, VAPID_SUB; the PEM is never staged.</done>
</task>
