<task type="auto">
  <name>Task 2: Subscribe/unsubscribe/vapid-key endpoints + client push.js</name>
  <files>web/push.py, web/urls.py, static/js/push.js</files>
  <action>
Create `web/push.py` (one module per surface). Endpoints, all `@login_required` and scoped to
request.user:
- `subscribe(request)`: `@require_http_methods(["POST"])`. Parse the JSON body
  (`{endpoint, keys:{p256dh, auth}}`). VALIDATE: endpoint is an https URL and keys has both p256dh
  and auth; return HttpResponseBadRequest otherwise (V5 input validation — threat T-05-14). Persist
  with `PushSubscription.objects.update_or_create(endpoint=endpoint, defaults={"user": request.user,
  "keys": keys})` (dedup on endpoint, RESEARCH Pitfall 6). Return JsonResponse ok. CSRF applies — the
  client sends X-CSRFToken explicitly (Pitfall 5).
- `unsubscribe(request)`: `@require_http_methods(["POST"])`. Delete the request.user's
  PushSubscription for the posted endpoint (only their own rows — T-05-09).
- `vapid_public_key(request)`: return the base64url settings.VAPID_PUBLIC_KEY as JSON (used by the
  client to build applicationServerKey).

Wire routes in web/urls.py importing web.push: `notifications/push/subscribe`,
`notifications/push/unsubscribe`, `notifications/push/key`.

Create `static/js/push.js` implementing the canonical flow (RESEARCH Pattern 3): `urlBase64ToUint8Array`,
`currentState()` (returns 'unsupported' | 'default' | 'granted'|'subscribed' | 'denied' from
Notification.permission + pushManager.getSubscription), and `enablePush(vapidPublicKey, csrftoken)`
which calls `Notification.requestPermission()` ONLY when invoked (from the Enable click, D-07), then
`reg.pushManager.subscribe({userVisibleOnly:true, applicationServerKey})` and POSTs sub.toJSON() to
/notifications/push/subscribe with the X-CSRFToken header. Banner logic: show the soft banner only
when currentState()==='default' and no localStorage "not-now" flag is set; "Not now" sets the flag
(re-askable from /notifications/settings); if 'denied', hide the banner and show the settings-page
passive "blocked — re-enable in browser settings" note (you cannot re-prompt a denied origin); if
'granted' without a subscription, silently re-subscribe. Populate the `#push-controls` placeholder on
the settings page. ASCII-only source comments.
  </action>

    <automated>py -3.12 manage.py test web.tests_push -v1</automated>
  </verify>
  <done>subscribe persists + dedups on endpoint, validates payload, is per-user; unsubscribe removes own rows; vapid key endpoint returns the public key; push.js drives the soft-prompt + subscribe flow.</done>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> subscribe endpoint | Client posts an endpoint + keys to be stored and later pushed to |
| client JS -> permission prompt | The origin's push permission is requested only via the gated Enable action |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-14 | Tampering | subscribe payload | high | mitigate | Validate endpoint is https + keys has p256dh/auth; reject otherwise; store only via update_or_create for request.user. Prevents junk/SSRF-shaped rows entering the outbox. |
| T-05-15 | Tampering | subscribe/unsubscribe CSRF | high | mitigate | Django CSRF; the raw fetch sends X-CSRFToken explicitly (Pitfall 5); POST-only via require_http_methods. |
| T-05-09 | Information Disclosure | unsubscribe / subscription rows | high | mitigate | Endpoints act only on request.user's PushSubscription rows; no cross-user endpoint deletion. |
| T-05-16 | Denial of Service (self) | permission "Block" | medium | mitigate | Soft pre-prompt (D-07) fires the browser prompt only on Enable; a stray Block cannot be triggered by an ignored/Not-now banner, keeping the origin re-askable. |
</threat_model>
