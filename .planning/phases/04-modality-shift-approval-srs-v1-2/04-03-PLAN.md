<objective>
Build the one genuinely-missing primitive of Phase 4: a pre-booking room-availability query. It answers "is room R free for [start,end)?" with half-open overlap semantics, building scope, Booking awareness, and — critically — request-awareness (an approved future ->F2F reservation occupies the room, D-18). This module powers the picker, the approval, and the materializer.

Purpose: D-08's "same conflict-check used for bookings" does not yet exist (booking conflict-check is Phase 7). Phase 4 builds the canonical availability primitive here so the picker (D-05/D-15), approval-time assignment (D-06), and born-assigned materialization (D-18) all share ONE definition of "free."
Output: ops/availability.py with five functions + a property-style RoomAvailabilityTests suite.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-02 | Tampering | room availability computation | high | mitigate | Availability is computed server-side from Session/Booking/approved-request IDs only — never from a client-supplied "is free" claim; the approval path (04-05) re-checks inside a transaction |
| T-04-05 | Tampering | request-aware reservation | high | mitigate | Approved ->F2F reservations occupy the room (D-18) so a future materialized session cannot silently take a reserved room; property tests cover the not-yet-materialized case |
| T-04-06 | Tampering | faculty double-book on time-move | high | mitigate | faculty_has_conflict blocks offering (and later server-rejects) a slot where the faculty already teaches (D-17) |
| T-04-HY010 | DoS | MSSQL single active result set | low | mitigate | list()-materialize candidate querysets before evaluation (pyodbc HY010 guard) |
