<objective>
Build the decision-consequence side of the modality-shift service: the transactional apply that turns an approved request into real room releases (->Online, MOD-03), real room assignments (->F2F/Blended, MOD-04), terminal denials when no room is free (D-07 REVISED), bundled time-moves (D-16/D-17), reservations for future sessions (D-18), and the decision/IFO notifications (MOD-05) — all-or-nothing, TOCTOU-safe, audited.

Purpose: This is where an approval becomes reality without breaking the effective-modality contract or double-booking a room. It composes the primitives from 04-03/04-04 inside one atomic transaction that re-checks availability at write time.
Output: scheduling/services.py apply functions + five test suites.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Dean client → approve service | Untrusted request pk + reason; a non-Dean or cross-department actor could try to approve |
| approval txn ↔ shared rooms | Two approvals could race for the same room between selection and write |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-03 | Tampering | TOCTOU room race | high | mitigate | Room re-resolved via ops/availability INSIDE transaction.atomic() at write time; consider select_for_update on candidate rooms; ApproveRaceTests proves re-resolution |
| T-04-01 | Elevation | approve authorization | high | mitigate | apply_approval requires dean.role==DEAN AND request.department==dean.department AND status==PENDING, re-checked inside the transaction; AuditLog per decision |
| T-04-02 | Tampering | forged room/modality at apply | high | mitigate | Room resolved server-side from schedule/building (never a client pk); declared_modality set from request.target_modality (validated at submit) |
| T-04-06 | Tampering | time-move double-book | high | mitigate | faculty_has_conflict re-checked at write; a forged double-booking slot forces DENY, not a partial apply (D-17) |
| T-04-07 | Tampering | partial apply on failure | high | mitigate | All-or-nothing: any no-room/deny rolls back every session write via transaction.atomic(); status becomes terminal DENIED, session state provably unchanged (D-07 REVISED) |
</threat_model>
