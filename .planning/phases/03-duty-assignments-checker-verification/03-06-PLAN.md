<objective>
Deliver CHK-08: offline checker scans queue locally in IndexedDB and, on reconnect, replay to a server endpoint that RE-VALIDATES each one against current state through the same pure gating core — applying it if still valid (preserving the original scan time) or recording it and flagging IFO if it no longer applies. Idempotent by a client-supplied uuid.

Purpose: A checker on a weak-signal floor must not lose scans, and the server must never blindly trust a stale offline decision — the room may have been handed over, ended, or already marked Absent between the offline scan and the replay.
Output: `static/checker/offline_queue.js` (vanilla IndexedDB queue + banner + reconnect replay), the `replay` endpoint in `web/checker.py`, the scan-page wiring, and the replay tests.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-19 | Tampering | trusting a stale offline decision | high | mitigate | Replay re-runs the pure core against CURRENT state, never the offline snapshot; non-actionable items are recorded + IFO-flagged, not applied (the CHK-08 "never blindly trusted" rule) |
| T-03-20 | Spoofing | forged replay batch from a non-on-duty user | high | mitigate | `checker_required` + CSRF; re-validation re-checks `_active_floor_ids(request.user, now)` + the room's CURRENT session state server-side per item — identical to the live `action`-view re-gate (03-02), so an off-duty or stale replay item is flagged/refused just like a live scan, never applied from the offline snapshot |
| T-03-21 | Tampering | duplicate/replayed uuid double-applying | medium | mitigate | Cache idempotency keyed on `client_uuid` (mirrors web/scan.py scan-idem); a repeated uuid returns `duplicate` |
| T-03-22 | Repudiation | backdated scanned_at abuse | medium | mitigate | Original scanned_at preserved for provenance but the DECISION uses current state + now; every applied/flagged item writes an AuditLog row |
| T-03-23 | Denial of Service | HY010 on the per-item write loop (MSSQL) | medium | mitigate | Items iterated over a materialized list (no `.iterator()`), each apply is a discrete write — the documented Pitfall 3 guard |
| T-03-SC | Tampering | package installs | low | accept | No new packages this phase; IndexedDB is browser-native (03-RESEARCH § Package Legitimacy Audit) |
