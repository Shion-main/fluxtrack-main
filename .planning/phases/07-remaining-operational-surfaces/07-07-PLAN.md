Views in `web/ifo.py` behind the existing `ifo_required` decorator — no new module, no new decorator.

`import_page` (GET-only) renders `templates/ifo/import.html`: an explanation of the flow, the upload form, and, when a staged-but-uncommitted file exists for this user, the preview panel for it so a page reload does not lose the review in progress. Before rendering, call `ops.import_staging.sweep_abandoned()` opportunistically. Plan 07-02 deliberately did not add a fifth scheduler job because the 4-job count is an asserted invariant (`NoImplicitSchedulerTests` / `SchedulerWiringTests`); sweeping here is the cheaper alternative for a surface used a handful of times per term. Say so in the docstring.

`import_preview` (POST-only) is the multipart entry point. Read the file from `request.FILES` and call `ops.import_staging.stage_upload(uploaded, request.user)`. Catch the staging module's error type and re-render the panel with the message at status 400 — the friendly-400 seam, matching `web/faculty.py:452`. On success, store ONLY the returned token in `request.session`; never the path and never the bytes. Then run the dry run against `ops.import_staging.staged_path(staging)`.

For the dry run, use both halves deliberately. Call `scheduling.importing.reconcile(data_rows, col)` DIRECTLY for the structured numbers — the four-bucket partition, the flagged unassigned/typo rooms, the distinct-room and instructor counts, and the email-less instructors — because the management command prints its reconciliation rather than returning it, and screen-scraping stdout for the primary display would be brittle. Read the rows through the command's own extension dispatch path so `.xlsx` and `.csv` are parsed by exactly the same code the CLI uses; do not write a second parser and do not add a parsing dependency (04.1-01 locked a stdlib zipfile/xml reader with no openpyxl or pandas). Additionally run `call_command("import_offerings", file=path, dry_run=True, stdout=io.StringIO())` and keep its captured output as a secondary detail/log pane, so the operator can see the command's own narrative.

Wrap the parse in a try that catches at minimum `zipfile.BadZipFile` (an `.xlsx` is a zip archive; a renamed non-zip raises here) and the generic parse failures, and render a friendly "this file could not be read as an offerings export" message. Checking the extension proves nothing about the content — the real validation is that the parser accepts it — and a garbage upload must be a message, not a 500.

Bound the parse. An `.xlsx` is a zip, so a crafted one can expand enormously and a cap on the uploaded bytes does not cap the expanded XML. Enforce a maximum parsed row count and refuse beyond it with a clear message. The surface is IFO-admin-only so exposure is limited, but this is the one input-validation gap worth naming.

`templates/ifo/import.html` extends `ifo/_console.html`. The upload form must carry `hx-encoding="multipart/form-data"` on the `<form>` element — htmx serializes as urlencoded by default, which cannot carry file content, so the file silently never arrives and the server sees an empty `request.FILES`. The `<input type="file">` needs a `name`, an `accept` listing the two extensions, and must sit inside the form htmx serializes. Include `{% csrf_token %}` inside the form so the token rides in the multipart body.

Target the htmx response at a SMALL status region, not at the whole form. `templates/base.html` configures `responseHandling` so a 400 swaps for inline validation, which is right for text fields but wrong for file fields: browsers forbid programmatic population of a file input, so a swapped-in `<input type="file">` is always empty and the user's selection is silently destroyed on every validation failure. Give the form a `hx-target` pointing at a small `#import-status` region with `hx-swap="innerHTML"`, so the form element and the user's chosen file survive a refusal. Add a `hx-indicator` spinner; upload progress UI is not required.

If any JavaScript is added to re-enable the submit control after a request, hang it off `htmx:afterRequest`, not `htmx:responseError`. The 400 rule in this project's `responseHandling` sets `"error": false`, so `htmx:responseError` does not fire for a 400 and such a handler would never run.

`templates/ifo/_import_panel.html` is the swapped partial holding the reconciliation report. Render the four-bucket counts as a summary with the balance identity visible (total rows equals intact plus roomless-TBA plus online-no-room plus no-schedule), so the operator can see at a glance that nothing was silently dropped. Render the flagged rooms and email-less instructors as tables using `.tbl` inside `.table-wrap`, paginated with `paginate` and `{% include "_pager.html" %}` if long. Use the console vocabulary (Franken UI `uk-*` plus Tailwind utilities); not the `.ft-*` navy vocabulary. Colour is never the only signal — every flag chip carries an icon and a text label. No border-left accent stripes. Use tokens from `static/css/tokens.css` for any colour; no hard-coded hex.

Only add CSS to `static/css/console.css` if the file-input control genuinely needs styling beyond `uk-input` — prefer reusing the existing vocabulary. If you do add any, keep it token-driven.

`import_commit` (POST-only, `ifo_required`). Read the staging token from `request.session` — never from a form field. A form-supplied token is an IDOR handle: it would let any IFO user commit any staged file by guessing or replaying a token. Resolve with `ops.import_staging.resolve_staged(token, request.user)`, which already filters on owner and unconsumed state. A None result renders a friendly "that upload is no longer available, please upload again" message at 400; it is not an exception.

Run the real apply: `call_command("import_offerings", file=staged_path, dry_run=False, ...)` with stdout captured, matching whatever term-name and sheet options the preview used so preview and commit can never describe different work. The apply is additive-only by construction (D-13) — `import_offerings` uses `get_or_create` throughout and deletes nothing, so re-running the same file is idempotent. Do NOT import `scheduling.management.commands.reset_term` anywhere in `web/`; the destructive term reset stays unreachable from the browser and its accepted cost (rows from a superseded file staying behind) is a recorded decision, not an oversight.

Add a Discard control POSTing to a `import_discard` view that calls `ops.import_staging.discard_staged` and clears the session key, so an operator who reviews a bad preview can walk away cleanly instead of leaving an orphan for the sweeper.

Wrap the apply so a mid-import failure renders a friendly error with the captured stdout rather than a 500. Do not consume the staging row on failure — the operator should be able to retry or discard.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser multipart body → server filesystem | Filename, extension, size and content are all attacker-controlled. |
| session token → staged file | Whoever presents a valid token addresses stored bytes. |
| staged file → bulk database write | The commit step writes hundreds of rows from that file. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-31 | Tampering | path traversal via `uploaded.name` | high | mitigate | Storage name built server-side from the token (plan 07-02); the client name is stored as display text and is never joined into a path. |
| T-07-32 | Information disclosure | committing another user's staged file | high | mitigate | Token read from `request.session` only, never from the form; `resolve_staged` filters on `uploaded_by`; asserted by the cross-user test. |
| T-07-33 | Denial of service | zip bomb in a crafted `.xlsx` | medium | mitigate | Upload size cap plus a maximum parsed row count; surface is IFO-admin-only. |
| T-07-34 | Denial of service | unreadable file producing a 500 | medium | mitigate | `BadZipFile` and parse failures caught and rendered as friendly messages; asserted by test. |
| T-07-35 | Denial of service | orphaned staged uploads filling the disk | medium | mitigate | `sweep_abandoned` called opportunistically on the import page; discard control offered explicitly. |
| T-07-36 | Tampering | destructive `reset_term` reachable from the browser | high | mitigate | D-13: no `web/` module imports it; RDS Express has no point-in-time restore configured, which raises the cost of any accidental mass delete. |
| T-07-37 | Elevation of Privilege | non-IFO reaching the import surface | high | mitigate | `ifo_required` on all four views; POST-only on the three write views; three-way authz plus 405 tests. |
| T-07-38 | Repudiation | untraceable bulk schedule import | medium | mitigate | `schedule.imported` AuditLog with the original filename, size and outcome counts. |
</threat_model>
