1. Move `_room_timetable` from `web/ifo.py:219` into `web/room_state.py` as public `room_timetable(room, term)`, joining `occupies` and `room_tile` there. Keep its docstring intact — it carries the reasoning for the campus-wide block ladder (a free slot must show as an empty cell rather than vanishing, and every room must print on the same grid so two printouts compare side by side) and the half-open slot-occupancy rule that makes a double-length class fill two rows with no rowspan bookkeeping. 2. Fix the wasted prefetch in `_guard_floor_ids` at `web/guard.py:44`. It calls `prefetch_related("floors")` and then `a.floors.values_list("pk", flat=True)` inside the loop — `values_list` on a related manager bypasses the prefetch cache and re-queries, so the prefetch buys nothing and the query count grows linearly with the guard's assignments. Replace the inner call with a comprehension over `a.floors.all()`, which does use the cache. Harmless today for a single user per poll; it matters at fan-out scale, and plan 07-12 builds its recipient resolution on the same shape, so fixing it here means that plan copies a correct pattern.

3. Align `_poll_ms` at `web/guard.py:53` with the rest of the codebase. It reads `settings.FLUXTRACK_POLICY["poll_interval_seconds"]` directly, while `web/ifo.py:179` reads the same value through `ops.policy.get_policy`. Those differ in a way that matters: `get_policy` honours `SystemSetting` overrides and the raw settings read does not, so an operator retuning the poll interval silently fails to affect the Guard monitor. Switch `_poll_ms` to `get_policy("poll_interval_seconds")`, matching Conventions §3 (policy-driven, never hardcoded and never bypassed). Record the change in the summary — it alters observable behaviour for anyone who has set an override.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Guard browser → room-by-code URL | The room code is client-supplied and enumerable. |
| duty assignment → what a Guard may see | On-duty status changes with the clock, so it must be re-derived per request. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-58 | Information disclosure | reading the schedule of a room on an unassigned floor | high | mitigate | Floor set re-derived server-side per request via `_guard_floor_ids`; off-floor room raises `Http404`, asserted by a test that the room code does not appear in the response. |
| T-07-59 | Information disclosure | room existence confirmed to an off-duty guard | medium | mitigate | 404 rather than 403, mirroring `web/checker.py:616`. |
| T-07-60 | Elevation of Privilege | a write reaching a Guard surface | medium | mitigate | `@require_http_methods(["GET"])` and the new URL added to `GuardReadOnlyTests`. |
| T-07-61 | Information disclosure | attendance status or flag history leaking onto a Guard surface | medium | mitigate | The page shows room state, schedule and faculty for the current class only; absence and flag history stay HR/Dean/IFO per D-07's minimum-access rule. |
| T-07-62 | Tampering | a second, drifting copy of the on-duty predicate | medium | mitigate | `assignment_covers_now` is called through `_guard_floor_ids`; no fourth copy is written. |
