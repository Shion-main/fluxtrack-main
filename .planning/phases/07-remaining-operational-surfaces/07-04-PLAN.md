  <files>web/ifo.py, web/urls.py, templates/ifo/room_rotate.html, templates/ifo/poster.html</files>
  <action>
Add two views to `web/ifo.py` behind the existing `ifo_required` decorator.

`room_rotate_confirm` is GET-only (`@require_http_methods(["GET"])`). Fetch the room by code with `get_object_or_404(Room.objects.select_related("floor__building"), code=code)` — the by-code lookup convention used everywhere else in this module. Render `templates/ifo/room_rotate.html` with the room and its current `code_rotated_at` / `code_rotated_by`, so the operator can see when it was last rotated and by whom.

`room_rotate` is POST-only (`@require_http_methods(["POST"])`). Inside `transaction.atomic()`: obtain a fresh credential pair from `campus.codes.new_room_credentials()` — the single minter plan 07-02 built, whose bounded retry is what keeps this destructive action from raising `IntegrityError` on the UNIQUE `manual_code` column. Never mint a code inline here: rotation is exactly the path where an intermittent 500 is worst, because it fires immediately before D-14 sends the operator to reprint, leaving them unsure whether the old poster is dead; assign `qr_token` and `manual_code`; stamp `code_rotated_at` with `timezone.now()` and `code_rotated_by` with `request.user`; save with `update_fields` naming exactly those four columns. Write an AuditLog with event type `room.code_rotated`, `target_type="room"`, `target_id` the room pk, and a payload recording the room code, the rotation instant, and the fact that a rotation occurred. Do NOT put the old or new token value in the audit payload — these are resolver-only credentials that are never rendered client-side (SCAN-07, §6.2), and an audit table is a much wider-read surface than the resolver.

On success `redirect("ifo_room_poster", code=room.code)` per D-14 — the flat-name redirect style already used at `web/ifo.py:287`, without `permanent`. Nothing is cached: the QR image is regenerated on demand from `room.qr_token` by the existing `room_qr` view, so rotation only has to change the stored values.

URL names `ifo_room_rotate_confirm` and `ifo_room_rotate`, registered in the existing IFO block of the flat `web/urls.py` list. Both paths sit under `ifo/rooms/<str:code>/` alongside the existing poster and QR routes.

`templates/ifo/room_rotate.html` extends `ifo/_console.html` and matches the confirm-page shape plan 07-03 established for room delete. It must state, naming the specific room code, that rotating invalidates the poster currently on that door and that the printed six-digit fallback code stops working at the same moment; it must say what to do next (print the new poster from the page it lands on and replace the one on the door); and it must show the last-rotated stamp if there is one. Use the console Franken UI vocabulary (`uk-card uk-card-body`, `uk-alert uk-alert-destructive` for the warning block with `role="alert"`, `uk-btn uk-btn-primary` for the confirm submit, a plain link back). No JavaScript confirm dialog — a GET confirm page is what this codebase gets, for the same keyboard and screen-reader reasons as the delete confirm. Colour is never the only signal: the warning carries an icon and a text label. No border-left accent stripes.

In `templates/ifo/poster.html`, add a small, non-printing notice showing the last-rotated stamp when present, so the operator arriving from a rotation sees confirmation that they are looking at the new credentials rather than a cached old page. Keep it out of the print layout.

Add a `RoomRotateTests(TestCase)` class to `web/tests_ifo_rooms.py`, next to the room CRUD classes from plan 07-03.

The "old credentials are dead" assertions matter more than the "new values differ" ones — capture both original values before the POST and assert afterwards that a `Room.objects.filter(qr_token=<old>)` lookup and a `Room.objects.filter(manual_code=<old>)` lookup both come back empty. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| IFO browser → resolver credentials | `qr_token` and `manual_code` are the only secrets standing between a passerby and a fraudulent check-in. |
| AuditLog table → readers | Audit rows are read far more widely than the resolver columns they might describe. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-15 | Information disclosure | audit payload containing rotated credentials | high | mitigate | Payload records the room and the instant only; asserted by a test that the payload contains neither old nor new values. |
| T-07-16 | Tampering | rotation via GET (link prefetch, crawler, accidental reload) | high | mitigate | `@require_http_methods(["POST"])` on the action, GET-only on the confirm page; asserted by 405 tests. |
| T-07-17 | Repudiation | untraceable credential rotation | medium | mitigate | `room.code_rotated` AuditLog with actor, plus the durable `code_rotated_at`/`code_rotated_by` stamps on the row. |
| T-07-18 | Denial of service | rotation producing a colliding code and 500ing | high | mitigate | Reuses `campus.codes.new_room_credentials` and its bounded collision retry (plan 07-02). Untreated, the observed ~2.3%-per-load collision rate makes this a real intermittent 500 on a destructive, audit-logged action. |
| T-07-19 | Denial of service | dead poster left on a door after rotation | medium | mitigate | D-14: confirmation names the consequence and the success path lands on the reprint page; poster shows the rotation stamp. |
</threat_model>
