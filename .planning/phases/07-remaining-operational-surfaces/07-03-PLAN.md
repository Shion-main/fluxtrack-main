<objective>
Give IFO a dedicated, non-admin room CRUD surface on the existing console, with a delete that refuses and NAMES what blocks it (D-17 as amended by D-19 and D-20) rather than surfacing a `ProtectedError` 500.

Purpose: rooms are currently creatable only through the Django admin or the importer. D-17 rules out soft-deactivate (an `is_active` field would have to be taught to every room query, including the scan path this phase leaves alone) and rules out cascade (it destroys attendance history). The refusal-with-names is therefore the whole feature, and plan 07-02 already built the probe and the database guarantee behind it.
Output: create / edit / delete-confirm console pages and `web/tests_ifo_rooms.py`. Credentials for a new room come from the shared minter plan 07-02 built; nothing is minted here.
</objective>

<task type="auto">
  <name>Task 2: Delete confirm page with named refusal</name>
  <files>web/ifo.py, web/urls.py, templates/ifo/room_delete.html</files>
  <action>
Add `room_delete` to `web/ifo.py` behind `ifo_required`, serving GET as the confirmation page and POST as the action. There is no destructive-confirm interaction anywhere in this codebase yet, so build it as a real GET page — not a JavaScript `confirm()` dialog, which cannot show the blocker detail and cannot be reached by keyboard-only or screen-reader users.

GET: fetch the room by code, call `campus.services.room_delete_blockers(room)`, and render `templates/ifo/room_delete.html`. When the dict is empty, the page states plainly that the room has no references and offers a Delete button. When it is non-empty, the page NAMES each blocking relation with its count in plain language — schedules, sessions, bookings, checker validations — and offers no Delete button at all, only a way back. Explain on the page why deletion is refused rather than offered-with-a-warning: deleting a referenced room would destroy attendance history, which is the one thing an attendance-integrity system may never do.

POST: re-run the probe inside `transaction.atomic()` and delete only if it is still empty — the GET-time probe is a display, never the authorization. A room can gain a session between the confirm page and the click, so this re-check is the actual control (the server-side re-gate convention this project applies to every action). Wrap the `.delete()` call in a `try` catching `django.db.models.deletion.ProtectedError` as a belt-and-braces backstop and render the same friendly refusal — the probes are for naming, PROTECT is for correctness, and neither alone is sufficient. Never let a `ProtectedError` reach the user as a 500.

Audit both outcomes. A successful delete writes `room.deleted` with the room code and floor in the payload. A refusal writes `room.delete_refused` with the blocker dict in the payload — D-17's discretion note explicitly lists the delete-refusal among the new audit events, and it gives IFO a trail of attempted destructive actions on rooms that turned out to be live.

Refusal renders at status 400 following the house 400-re-render contract; success redirects to `ifo_rooms`. URL name `ifo_room_delete`, path under the existing IFO block.

`templates/ifo/room_delete.html` extends `ifo/_console.html`, uses the console Franken UI vocabulary, and presents the blocker list as a simple table using `.tbl` inside `.table-wrap` from `static/css/app.css`. Colour is never the only signal — each blocker row carries an icon and a text label alongside any colour treatment (WCAG 1.4.1, the rule stated at `web/checker.py:636`).

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-10 | Tampering | `room_delete` POST | high | mitigate | Re-run `room_delete_blockers` inside `transaction.atomic()` at POST time; the GET probe is display only. `ProtectedError` caught as backstop. |
| T-07-11 | Denial of service | non-numeric floor/capacity reaching the ORM | medium | mitigate | Format validation before any ORM write, per the CR-04 ordering rule at `web/ifo.py:375`; bad input is a friendly 400. |
| T-07-12 | Denial of service | credential collision on room create | medium | mitigate | Credentials sourced only from `campus.codes.new_room_credentials` (plan 07-02), which retries on collision instead of letting `IntegrityError` become a 500. No inline minting here. |
