<objective>
Close GRD-04: debounced web-push alerts for floor activity, coalesced to one notification per on-duty Guard per sweep run.

Purpose: this is the riskiest requirement in the phase because it touches the sweep — the most safety-critical, most guarded code path in the project — so it is sequenced last and designed to be strictly ADDITIVE. The `wave: 10` here is a deliberate risk-ordering choice with no logical dependency on any other Phase 07 plan; it can run in parallel with plan 07-11, whose files are disjoint from these.

Two design constraints are settled before implementation begins. D-06 puts the debounce at the job cadence: both triggers already fire inside the same five-minute sweep run, so one push per guard per run needs no timer, no `last_alerted_at` column and no new policy knob. And the integer returns of `sweep_no_shows` and `detect_room_conflicts` are load-bearing at fourteen call sites — ten test assertions plus both command wrappers, including a test that documents itself as the guard a future sweep edit must not break — so those signatures gain an optional collector and keep their scalar returns exactly as they are.
Output: a registered notification type, a collector seam, a recipient/summary module, caller-side coalescing, and tests.
</objective>

<task type="auto">
  <name>Task 2: Additive collector on the sweep functions + ops/guard_alerts.py</name>
  <files>scheduling/jobs.py, ops/guard_alerts.py</files>
  <action>
Collector seam. Give both sweep functions an OPTIONAL keyword argument defaulting to None — `sweep_no_shows(now=None, collect=None)` and `detect_room_conflicts(now=None, collect=None)` — and leave their scalar integer returns exactly as they are. Every existing call site, every assertion on the return value, and both command wrappers keep working untouched. This is non-negotiable: the returns are asserted at ten places across `scheduling/tests.py` and `scheduling/tests_merge_sweep.py`, and the latter documents itself as the guard that a future edit to `sweep_no_shows` must not break. Changing the return type to a tuple or a dataclass would break all of that and would read as a regression in exactly the area Phase 2 built guards around.

When `collect` is not None, each function appends a `(kind, floor_id)` tuple for each event it produces — a short kind string distinguishing a swept-absent room from a newly-flagged conflict, so the summary text can describe them differently.

Read the floor id off data the loop ALREADY has. In `sweep_no_shows`, add `"room"` to the existing `select_related` on the candidate queryset so `s.room.floor_id` costs nothing extra, and append inside the existing per-session loop after the status flip commits. In `detect_room_conflicts`, the per-conflict block already fetches the `Room` for its label; append `room.floor_id` there. Do NOT build a room-id list and look up floors afterwards. `sweep_no_shows` backfills all past-date no-shows and self-heals after an outage, so after a multi-day scheduler gap on a 2113-schedule term a single run can mark hundreds or low thousands of sessions absent — a `pk__in` over that batch is a real exposure to the MSSQL 2100-parameter limit, the same failure class as the 04.1-04 `reset_term` incident, and it would only ever appear after downtime.

Do NOT touch the existing per-conflict `notify(role=Role.IFO_ADMIN, type="room_conflict", ...)` call. IFO wants one notification per conflict; changing it would alter Phase 2 behaviour and break its tests. And do NOT put the guard notification inside either loop — that would be one push per event, the exact spam D-06 forbids.

New module `ops/guard_alerts.py` holding the recipient resolution and the summary, so `scheduling/jobs.py` gains no notification logic and the resolution is unit-testable without running a sweep.

`notify_floor_guards(events, now=None)` takes the collected `(kind, floor_id)` tuples and returns the number of guards notified. Shape:
  - Return 0 immediately on an empty collection.
  - Reduce the events to the set of affected floor ids and a per-floor tally by kind.
  - Load every active FLOOR-scoped GUARD `Assignment` in ONE query with `select_related("user")` and `prefetch_related("floors")`, and materialize it with `list()`. The query filters on scalar equality only — no `IN` list, no exposure to the parameter limit. Materializing matters because this function writes `Notification` rows while iterating, and pyodbc runs with MARS off, so writing inside a live cursor raises HY010; both existing sweep functions carry the same guard.
  - Read each assignment's floors with a comprehension over `a.floors.all()`, NOT `values_list`, which bypasses the prefetch cache and re-queries per assignment.
  - Skip an assignment whose user is not `is_active`. `notify(role=...)` filters active users but `notify(users=[...])` does not — it trusts the caller — so passing a deactivated guard would create a notification for a disabled account.
  - Decide on-duty with `verification.resolver.assignment_covers_now(a, today, now_t)` on the LOCAL date and time via `timezone.localtime`. Do not write a fourth copy of that predicate; its docstring exists so the three existing copies can never drift again.
  - Intersect each on-duty guard's floors with the affected floor set; guards with no overlap get nothing.
  - Emit ONE `notify(users=[guard], type=<the registered constant>, title=<summary>, body=<detail>, link="/guard/monitor")` per remaining guard. `notify()` is the single Notification write path (NOTIF-00, enforced by a source-grep test in `ops/tests.py`); never construct a `Notification` directly.

Write a small pure summary helper in the same module turning a per-floor tally into readable text in the shape D-06 asks for — for example rooms now free on a floor, and conflicts detected on a floor, naming the floor. Keep it pure and unit-test it separately from the notification path.

Guard-population note: `notify(users=[...])` creates rows in a Python loop, an N+1 by construction, but N is the number of on-duty guards and is single-digit. Do not optimize it — that would mean editing the single-write-path function every other feature depends on. Similarly, do not touch `send_push_outbox` in this plan; its per-row `muted_types` and subscription lookups are a known N+1 that the D-06 coalescing is precisely what keeps bounded.
  </action>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scheduled job → notification recipients | The recipient list is derived from duty data, never from any request. |
| notification content → Guard device | A push renders on a locked screen. |
| sweep batch size → database parameter limits | Batch size is unbounded after an outage. |

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
