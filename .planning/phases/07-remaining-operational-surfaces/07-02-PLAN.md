<objective>
Build the data-layer foundations the rest of Phase 07 stands on, in one migration so ops migration numbering stays unambiguous: (a) make the room-delete refusal a database guarantee and give the UI a probe that can name each blocker, (b) fix a live intermittent uniqueness defect in room-code minting that two later plans in this phase would otherwise turn into user-facing 500s, and (c) build the upload staging mechanism that holds an import file between preview and commit.

Purpose — three findings, all model-layer, all settled before any surface is built:
  - Research found `ops.Booking.room` is `on_delete=CASCADE` while every other Room FK is `PROTECT`. As written, D-17's Booking blocker was unenforceable and a room whose only references were bookings would delete cleanly and destroy them (D-19).
  - A verified baseline suite run produced `pyodbc.IntegrityError: Violation of UNIQUE KEY constraint 'UQ_campus_room_manual_code'` in `scheduling.tests_room_master.LoadRoomMasterTests`. Room manual codes are minted with no uniqueness pre-check and no retry, and at the real term scale (~218 rooms in a six-digit space) that is roughly a 2.3% collision chance per full load. It is intermittent, which is why it has been passing as one of the "pre-existing failures" — it is a live defect, not a stable known failure. Plan 07-04 mints a fresh code on this same path for a destructive, audit-logged rotation, and plan 07-07 wraps the importer in a web upload, so both would inherit an intermittent 500.
  - There is no `CACHES` and no `SESSION_ENGINE` setting in this project, so neither the cache nor the session can hold a multi-MB `.xlsx` between two requests.
Output: one ops migration, an `ImportStaging` model, a pure staging service, a four-relation room-delete probe, and a collision-safe code minter.
</objective>

Two model changes in `ops/models.py`, then ONE generated migration covering both. Both live in the same migration deliberately: two plans in this phase would otherwise race for the `0005_` slot.

Change 1 (D-19). `Booking.room` is currently `on_delete=models.CASCADE`. Change it to `models.PROTECT`, keeping `related_name="bookings"`. Add a field comment explaining why: every other FK targeting `campus.Room` (`scheduling.Schedule.room`, `scheduling.Session.room`, `verification.CheckerValidation.room`) is already PROTECT, and D-17 requires a room delete to be refused when bookings reference it. Under CASCADE that refusal was a view-level courtesy that silently destroyed booking history when bypassed; under PROTECT the database itself is the guarantee. Note in the comment that this changes behaviour for the existing Django-admin Booking surface too (a room delete from admin now raises `ProtectedError` instead of cascading) — that is the intended effect of D-19, not a regression.

Change 2 (D-12). Add an `ImportStaging` model to `ops/models.py` holding a `.xlsx`/`.csv` between the IFO-03b dry-run preview and the commit. Fields: `token` (CharField max_length=64, unique, db_index — an opaque `secrets.token_urlsafe(32)` value, never user-supplied), `uploaded_by` (FK to `settings.AUTH_USER_MODEL`, `on_delete=CASCADE`, `related_name="import_stagings"`), `original_name` (CharField max_length=255 — display only; explicitly documented as never used to build a path), `stored_path` (CharField max_length=255 — storage-relative and server-built), `size_bytes` (PositiveIntegerField), `created_at` (auto_now_add), `consumed_at` (nullable DateTimeField). Meta ordering newest-first. The model docstring must state the design contract: the bytes live on disk under MEDIA_ROOT, the session carries only the token, and this row is the ownership plus lifecycle record. Explain in the docstring why bytes are not in the session (the MSSQL `django_session.session_data` text column would be re-read on every authenticated request for the life of the session) and not in the cache (the default `LocMemCache` is per-process and would lose the file between the preview worker and the commit worker).

Generate the migration with `makemigrations ops` and then rename the produced file to `0005_booking_protect_import_staging.py`, updating its internal class only if the tooling embedded a name. Follow the 04-01 precedent for renaming a migration to a plan-chosen artifact name: unapply if already applied, rename, re-apply. Run `sqlmigrate` on it and record in the summary whether the `Booking.room` AlterField emits real SQL on mssql-django — Phase 01 found `AlterField` can be a no-op on this backend, and if the FK constraint is not actually rewritten, the PROTECT behaviour is Django-level only and the summary must say so plainly.

    - A Room with no references returns an empty dict.
    - A Room referenced by an active Schedule returns a dict containing a schedules count.
    - A Room referenced by a Session returns a dict containing a sessions count.
    - A Room referenced by a Booking returns a dict containing a bookings count — including a booking whose status is not "active" (D-19: CASCADE would have destroyed it, so it counts).

    - generate_manual_code returns a six-digit string.
    - When the random source yields a value already taken by an existing Room, generate_manual_code discards it and returns a different, unused code — no IntegrityError escapes.
    - When the random source yields taken values for every permitted attempt, generate_manual_code raises a clear domain error rather than returning a colliding code.
    - Creating two rooms in a row while the random source repeats its first value yields two rooms with distinct manual codes.
  </behavior>
  <action>
Two small pure modules in the `campus` app, both with tests in `campus/tests.py`. They share a task because they share a shape: pure helpers over the Room table with no module-level cross-app imports.

**Part A — `campus/services.py` with `room_delete_blockers(room)`,** returning a dict mapping relation name to count, containing only the non-zero entries — an empty dict means the room is genuinely unused and safe to delete.

Implement it with four reverse-accessor counts off the passed room instance: `room.schedules.count()`, `room.sessions.count()`, `room.bookings.count()`, `room.validations.count()`. Use the reverse accessors specifically so this module imports no model from `scheduling`, `ops` or `verification` — `campus` is the lowest app in the dependency order and must not gain upward imports.

Count ALL bookings, not just `status="active"` (D-19). The docstring must say why: D-10 treats a cancelled booking as not occupying the room for availability purposes, but a cancelled booking is still historical data that the pre-D-19 CASCADE would have destroyed, so "referenced by any Booking" is read conservatively here.

The docstring must also record the three rejected alternatives and why, so a later reader does not "improve" this into something worse: `django.contrib.admin.utils.NestedObjects` is an undocumented admin internal that loads thousands of model instances to answer a yes/no question; `django.db.models.deletion.Collector` resolves the whole deletion graph and signals blockers by raising, which is exception-driven control flow for a question the UI needs as data; four scalar-predicate counts on indexed FK columns are cheap, build no parameter list (2100-param safe), and return exactly the shape the UI needs to name blockers.

**Part B — `campus/codes.py` with `generate_manual_code()`,** the single minter of `Room.manual_code`.

`campus/models.py:37` declares `manual_code` as a six-character UNIQUE column with case-sensitive collation. Today it is minted at `scheduling/management/commands/load_room_master.py:140` and again at `scheduling/management/commands/import_offerings.py:254`, both as a bare formatted `secrets.randbelow(1000000)` inside a `get_or_create` defaults dict, with no uniqueness pre-check and no retry anywhere. A verified suite run has already produced `pyodbc.IntegrityError: Violation of UNIQUE KEY constraint 'UQ_campus_room_manual_code'` from `LoadRoomMasterTests`. This is a live intermittent defect.

Implement the minter as a bounded retry loop — around ten attempts — generating a candidate, skipping it when `Room.objects.filter(manual_code=candidate).exists()`, and raising a clear domain error if every attempt collides. Import `Room` lazily inside the function so the module keeps its no-module-level-cross-app-import property. The docstring must record the arithmetic that makes this necessary rather than paranoid: roughly 218 rooms in a six-digit space gives about a 2.3% chance of at least one collision per full term load, which is exactly the frequency that produces an intermittent failure nobody can reproduce on demand.

Also expose `new_room_credentials()` in the same module, returning a `(qr_token, manual_code)` pair, with the token from `secrets.token_urlsafe(24)` and the code from `generate_manual_code()`. Note in its docstring that the token needs no retry — 192 bits of entropy makes collision negligible — so the retry effort is deliberately spent only where the birthday arithmetic actually bites. Plan 07-03 (room create) and plan 07-04 (IFO-02 rotation) both consume this function and must not mint codes themselves.

Then change BOTH existing call sites to call `generate_manual_code()` (or `new_room_credentials()`) instead of formatting a raw random value. That is the point of extracting it: the importer, the room-create form and the rotation view can never disagree about how a code is minted, the same one-shared-atom convention this project already applies to `is_no_show_past_grace` and `csv_safe`.

**Tests.** Add `RoomDeleteBlockerTests` and `ManualCodeMintingTests` to `campus/tests.py`, covering all eleven behaviours. Build fixtures with method-local imports of `scheduling`/`ops`/`verification` models (the `ops/tests.py:128` idiom) so `campus/tests.py` stays import-light, and reuse `scheduling.tests.make_session` for the FK chain where a Session is needed.

The collision tests must be DETERMINISTIC, not probabilistic. Patch the module's random source so it returns a known already-taken value on the first call or calls, then a distinct free value, and assert the returned code is the free one and that no `IntegrityError` escapes. A test that simply creates many rooms and hopes for a collision reproduces the original bug's worst property — it fails only sometimes.

  <name>Task 3: ops/import_staging.py — stage, resolve, consume, discard, sweep</name>
  <files>ops/import_staging.py, ops/tests_staging.py</files>
  <behavior>
    - `stage_upload` on a valid `.xlsx` writes a file under the staging prefix, creates an ImportStaging row, and returns it with a token.
    - The stored path is derived from the token and the extension only — a hostile `original_name` such as one containing directory separators or parent-directory segments does not influence the path, and the original name is preserved verbatim in `original_name` for display.
    - `stage_upload` refuses an extension outside the allowed set and refuses a file over the size cap, raising the module's error type with a user-safe message — never a 500.
    - `resolve_staged` returns the row for the correct owner and returns None for a different user, for an unknown token, and for an already-consumed row.
    - `consume_staged` stamps `consumed_at`; a second call on the same row resolves to None.
    - `discard_staged` deletes the file and the row.
    - `sweep_abandoned` removes rows older than the TTL whose `consumed_at` is NULL, deleting the file before the row, and leaves consumed and fresh rows alone.
    - A row whose underlying file is already missing is swept without raising.
  </behavior>
  <action>
Create `ops/import_staging.py` as the pure service layer for D-12's preview-then-commit hold. No HTTP, no `request` object — it takes an uploaded-file object and a user, so it unit-tests without a client.

Module constants: `ALLOWED_EXTENSIONS` as the set of `.xlsx` and `.csv` (D-12 — `import_offerings` already dispatches by extension and `.xlsx` is what IFO actually has); `MAX_UPLOAD_BYTES` as an explicit cap; `STAGING_PREFIX` as the storage-relative directory under MEDIA_ROOT; `STAGING_TTL_HOURS` as a module constant of 2. Document that the TTL is a module constant and deliberately NOT a `FLUXTRACK_POLICY` knob — it is not an operator-tunable business rule, mirroring the `_WINDOW_MINUTES` precedent in `ops/push.py`. Note in the docstring that neither `DATA_UPLOAD_MAX_MEMORY_SIZE` nor `FILE_UPLOAD_MAX_MEMORY_SIZE` imposes a maximum upload size — the former governs the non-file body and the latter governs only where the file lands — so `MAX_UPLOAD_BYTES` checked against `uploaded.size` is the only cap that exists.

Define a module error type (e.g. `ImportStagingError`) carrying user-safe messages, mirroring the `ModalityShiftError` friendly-400 seam that `web/faculty.py:452` consumes.

`stage_upload(uploaded, user)`: validate extension (lowercased, from the client name, used ONLY for the allow-list decision and the stored suffix) and `uploaded.size` against the cap; generate a token with `secrets.token_urlsafe(32)`; build the storage-relative name entirely server-side as the staging prefix plus the token plus the extension; write it through `default_storage.save` streaming `for chunk in uploaded.chunks()` — never `uploaded.read()`, and never `uploaded.temporary_file_path()`. Create and return the `ImportStaging` row.

The docstring must call out the single most likely bug in a naive D-12 implementation: Django unlinks the upload temp file when the request finishes, and a sub-2.5 MB upload arrives as an `InMemoryUploadedFile` that never had a path at all, so stashing `temporary_file_path()` for the commit request fails — and fails differently for the small `.csv` fixture than for the real multi-MB `.xlsx`. Writing our own copy in the preview request is what makes the two-request flow work.

`resolve_staged(token, user)`: return the row filtered by token AND `uploaded_by=user` AND `consumed_at__isnull=True`, or None. Docstring states the ownership filter is the IDOR control and that callers must pass the token read from the session, never from a form field.

`consume_staged(staging)`: stamp `consumed_at` and save with `update_fields`.
`discard_staged(staging)`: delete the stored file via `default_storage.delete` guarded so a missing file is not an error, then delete the row.
`staged_path(staging)`: return the absolute filesystem path via `default_storage.path(...)` — the commit step needs a real path because `import_offerings` reads `--file <path>` from the filesystem.
`sweep_abandoned(now=None)`: delete file-then-row for unconsumed rows older than the TTL, returning the count. Materialize the queryset with `list()` before deleting inside the loop — MSSQL HY010, the same guard both sweep functions in `scheduling/jobs.py` carry.

Add `ops/tests_staging.py` with the eight behaviours above. Use `override_settings(MEDIA_ROOT=<tmp dir>)` so tests never write into the repo's `media/`. Build both upload shapes: a `SimpleUploadedFile` for the small case and one deliberately larger than 2.5 MB so the `TemporaryUploadedFile` handler path is exercised too — the two handlers behave differently enough that testing only the small fixture proves nothing about the real file.

Do NOT wire a scheduler job for `sweep_abandoned` in this plan. The 4-job scheduler count is an asserted invariant (`NoImplicitSchedulerTests` / `SchedulerWiringTests`); plan 07-06 decides whether to call the sweep opportunistically from the upload view instead of adding a fifth job, and owns any test-count update if it does add one.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| uploaded file → server filesystem | The client controls the filename, the extension, the size and every byte. |
| session token → staged row | Whoever holds the token addresses a stored file. |
| view-level policy → database constraint | A refusal implemented only in a view is bypassable by admin, shell, or a future caller. |
