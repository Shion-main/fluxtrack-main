<objective>
Build the data-layer foundations the rest of Phase 07 stands on, in one migration so ops migration numbering stays unambiguous: (a) make the room-delete refusal a database guarantee and give the UI a probe that can name each blocker, (b) fix a live intermittent uniqueness defect in room-code minting that two later plans in this phase would otherwise turn into user-facing 500s, and (c) build the upload staging mechanism that holds an import file between preview and commit.

Purpose â€” three findings, all model-layer, all settled before any surface is built:
  - Research found `ops.Booking.room` is `on_delete=CASCADE` while every other Room FK is `PROTECT`. As written, D-17's Booking blocker was unenforceable and a room whose only references were bookings would delete cleanly and destroy them (D-19).
  - A verified baseline suite run produced `pyodbc.IntegrityError: Violation of UNIQUE KEY constraint 'UQ_campus_room_manual_code'` in `scheduling.tests_room_master.LoadRoomMasterTests`. Room manual codes are minted with no uniqueness pre-check and no retry, and at the real term scale (~218 rooms in a six-digit space) that is roughly a 2.3% collision chance per full load. It is intermittent, which is why it has been passing as one of the "pre-existing failures" â€” it is a live defect, not a stable known failure. Plan 07-04 mints a fresh code on this same path for a destructive, audit-logged rotation, and plan 07-07 wraps the importer in a web upload, so both would inherit an intermittent 500.
  - There is no `CACHES` and no `SESSION_ENGINE` setting in this project, so neither the cache nor the session can hold a multi-MB `.xlsx` between two requests.
Output: one ops migration, an `ImportStaging` model, a pure staging service, a four-relation room-delete probe, and a collision-safe code minter.
</objective>
