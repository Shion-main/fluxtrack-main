<objective>
Give `ops.Booking` — a model that has existed since Phase 1 with only a Django-admin surface — a real IFO console UI, conflict-checked by the single canonical occupancy oracle.

Purpose: IFO-05's "conflict-checked" means exactly one thing (D-08): call `ops.availability.room_is_free()`, the same function the faculty room picker and Dean approval already use. It already counts active Bookings, room-holding non-Online Sessions, and approved modality-shift reservations with half-open overlap. Inventing a second conflict definition in the web layer is the specific failure this plan exists to avoid. Cancellation is decided by existing code too (D-10): `room_is_free` counts only `status="active"` bookings, so cancel is a status flip and the room frees itself.

Note on sequencing: `depends_on` here is file contention on `web/ifo.py`, `web/urls.py` and `web/tests_ifo_ops.py` with plan 07-05, not a logical dependency.
Output: booking list, create form with conflict refusal, cancel action, and tests.
</objective>

`booking_create` follows the `assignment_create` write shape at `web/ifo.py:342` including its deliberate ordering rule stated at `web/ifo.py:375`: validate FORMAT before any ORM write. A `parse_date` returning None, a `parse_time` returning None, or a non-numeric room pk fed to the ORM raises an unhandled `ValidationError` at INSERT time and becomes a 500 (CR-04). The validation ladder, in order:
  1. occupant name present and within the model's length;
  2. room pk numeric and resolving to a real Room;
  3. date parses, start time parses, end time parses;
  4. end strictly after start (a zero-length or inverted window is an operator error, and half-open overlap semantics make a zero-length booking occupy nothing);
  5. only then combine date plus times into aware datetimes with `timezone.make_aware(datetime.combine(...))`, matching the construction used in `ops/availability.py:106`, so a booking and a reservation are compared on the same clock.

The conflict check is one call: `room_is_free(room, start, end)` from `ops/availability.py`. If it returns False, refuse. Do NOT write a second overlap query anywhere in `web/ifo.py` and do NOT modify `ops/availability.py` — its half-open semantics live in `_session_occupants` and `_reservation_occupies` and are the single oracle (D-08).

The refusal message must be useful: state that the room is not free for that window and point the operator at the room's existing schedule (the room panel or room detail page for that room code) so they can see what occupies it. Deriving and rendering the specific occupant is optional; if implemented, derive it with the SAME queryset shapes `ops/availability.py` uses rather than a fresh guess, and materialize with `list()` before follow-up queries (MSSQL HY010).

No override control. D-09's parenthetical about an override describes the default refusal, and nothing in IFO-05 asks for a way to double-book over a scheduled class. Building one would let the console create precisely the contradictory occupancy that JOB-02c exists to flag. State this reading in the view docstring and record it in the plan summary so the operator can correct it if they disagree.

On success create the `Booking` with `created_by=request.user` and `status="active"`, then write an AuditLog `booking.created` with the room code, window and occupant in the payload (the `noun.verb_past` convention at `web/ifo.py:403`). Refusal re-renders `templates/ifo/_booking_form.html` at status 400 with the error, matching the house 400-re-render contract.

URL names `ifo_bookings` and `ifo_booking_create`, in the existing IFO block of the flat `web/urls.py` list.

| Boundary | Description |
|----------|-------------|

| web layer → occupancy truth | A second conflict definition in the web layer would silently disagree with the scan and approval paths. |

|-----------|----------|-----------|----------|-------------|-----------------|
