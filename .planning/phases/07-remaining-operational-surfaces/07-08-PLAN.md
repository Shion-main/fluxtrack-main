  <name>Task 1: accounts/photos.py — validate, decode, re-encode</name>
  <files>accounts/photos.py, accounts/tests_photos.py</files>
  <behavior>
    - A valid small JPEG returns non-empty bytes that Pillow can reopen, whose format is JPEG.
    - A valid PNG is accepted and the returned bytes are JPEG (converted), with no alpha channel.
    - An image larger than the target dimension is downscaled and its aspect ratio is preserved; an image smaller than the target is not upscaled.
    - A JPEG carrying EXIF (including a GPS tag) returns bytes with no EXIF present.
    - An image whose EXIF says it is rotated comes back with the rotation applied to the pixels.
    - A truncated or structurally corrupt image raises the module error with a user-safe message.
    - A non-image file (e.g. plain text with a .jpg name) raises the module error.
    - A GIF or BMP — a real, decodable image outside the allowed set — is refused on format, proving the allow-list is enforced on the decoded format and not on the filename.
    - An image whose declared pixel count exceeds the module's limit raises rather than decoding.
    - A file over the byte cap raises before any decode is attempted.
  </behavior>
  <action>
Create `accounts/photos.py` holding `normalize_profile_photo(fileobj)`, a pure bytes-in/bytes-out function that unit-tests without HTTP. Pillow is already a dependency (`Pillow>=10.0`); add no packages.

Module constants: `MAX_UPLOAD_BYTES`, `ALLOWED_FORMATS` as the set of JPEG and PNG, and `TARGET` as the standard output dimension. Define a module error type carrying user-safe messages, feeding the project's friendly-400 seam.

The pipeline order matters and each step earns its place:
  1. Check `fileobj.size` (or the byte length) against the cap FIRST, before any Pillow call. Note in the docstring that neither `DATA_UPLOAD_MAX_MEMORY_SIZE` nor `FILE_UPLOAD_MAX_MEMORY_SIZE` caps an upload — the first governs the non-file body, the second only decides where the file lands — so this explicit check is the only size limit that exists.
  2. `seek(0)`, `Image.open`, then `verify()` as a cheap structural pre-filter.
  3. **Reopen.** `verify()` closes the image and Pillow's documentation is explicit that the file must be reopened before any further use. This is the step that gets skipped, and skipping it is silent in a naive test: code that verifies and then only reads `format` and `size` appears to work while never decoding anything, so a truncated file sails through validation and fails later at render time. `seek(0)` and `Image.open` again.
  4. Check the decoded `format` against `ALLOWED_FORMATS`. Never trust the uploaded filename's extension or the browser-supplied `content_type`; both are attacker-controlled.
  5. Apply `ImageOps.exif_transpose` BEFORE stripping metadata, so a phone photo taken sideways has its rotation baked into the pixels rather than being lost with the tag it depended on.
  6. `convert("RGB")` — this is what actually forces a full decode. `Image.open` is lazy and reads the header only; without a decode-forcing operation a file with a valid header and a corrupt body passes. It also drops alpha and palette modes so the JPEG save cannot fail on mode.
  7. `thumbnail(TARGET, Image.LANCZOS)` — aspect-preserving and never upscaling. A fixed-tuple `resize` would distort a non-square photo. If a square Checker view is wanted, centre-crop before resizing; that is a discretionary UI call — pick one, and say which in the docstring.
  8. Save to a `BytesIO` as JPEG with a quality setting and `optimize=True`. Do NOT pass an `exif=` keyword: Pillow only writes EXIF when asked, so omitting it drops orientation, timestamps, camera model and GPS coordinates. Return `getvalue()`.

Decompression-bomb handling needs both controls, because a byte-size cap does not help at all — the entire point of a bomb is that a small file decodes to an enormous bitmap. Lower `Image.MAX_IMAGE_PIXELS` for this path to a value appropriate to a profile photo, and escalate `Image.DecompressionBombWarning` to an exception inside a `warnings.catch_warnings()` block scoped to the decode — between the limit and twice the limit Pillow only warns, and the decode proceeds and eats the memory. Catch `Image.DecompressionBombError` alongside `OSError` and `UnidentifiedImageError`.

The docstring must state plainly that the re-encode is the security control: the stored bytes are bytes Pillow produced, so polyglots, appended archives and smuggled metadata all die at once, and that `ImageField`'s own `to_python()` validation is a floor rather than a ceiling — it does not enforce a format allow-list, a size cap, or bomb limits.

Create `accounts/tests_photos.py` with `PhotoValidationTests`, `PhotoReencodeTests` and `DecompressionBombTests` covering the ten behaviours. Generate fixtures in-memory with Pillow rather than committing binary files; build the corrupt case by truncating valid JPEG bytes.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| faculty browser multipart body → media storage | Every byte, the filename and the declared content type are attacker-controlled. |
| stored media → Checker's browser | The photo is rendered to another user as identity evidence. |
| request → target user | The upload must act on the session's own user, never a supplied id. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-39 | Tampering | polyglot or payload-carrying image | high | mitigate | Full Pillow re-encode; the stored bytes are Pillow's output, asserted by a test that the stored file reopens as JPEG. |
| T-07-40 | Information disclosure | EXIF GPS coordinates from a phone photo | high | mitigate | `exif=` omitted on save so EXIF is dropped; `exif_transpose` applied first so rotation is not lost; asserted by test. |
| T-07-41 | Denial of service | decompression bomb | high | mitigate | `MAX_IMAGE_PIXELS` lowered for this path AND `DecompressionBombWarning` escalated to an error in a scoped `catch_warnings`; a byte cap alone does not help. |
| T-07-42 | Spoofing | format allow-list bypassed via filename or content_type | medium | mitigate | Allow-list checked against the DECODED format, never the extension or the browser-supplied type; asserted by the GIF/BMP refusal test. |
| T-07-43 | Elevation of Privilege | changing another user's identity photo | high | mitigate | Target is always `request.user`; any posted user id is ignored; asserted by the IDOR test. |
| T-07-44 | Denial of service | storage filling with collision-suffixed orphans | medium | mitigate | Old name captured before save and deleted after, guarded for a missing file; asserted by the orphan test. |
| T-07-45 | Tampering | content sniffing of user-uploaded media | medium | mitigate | `SECURE_CONTENT_TYPE_NOSNIFF = True` set explicitly rather than relying on an unverified default. |
| T-07-46 | Repudiation | untraceable identity-photo change | low | mitigate | `user.photo_updated` AuditLog carrying the user pk and byte size, no image data. |
</threat_model>
