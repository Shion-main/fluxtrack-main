<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| web worker vs scheduler process | Jobs must run in exactly one dedicated process; a scheduler embedded in web workers double-fires (duplicate absences/notifications) |
| unattended job execution | A crashing job must not kill the scheduler or pass silently; failures must be recorded and surfaced |
| dependency supply chain | An accidental APScheduler 4.0 jump would break the 3.x jobstore/scheduler API mid-milestone |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-12 | Tampering (duplicate state) | scheduler placement | high | mitigate | BlockingScheduler started ONLY by runscheduler (its own process); NoImplicitSchedulerTests asserts no AppConfig.ready starts one; max_instances=1 + coalesce prevent self-overlap |
| T-02-13 | Denial of Service | job failure crashing the process | high | mitigate | run_job catches broad Exception, records status=failed, never re-raises — one bad run cannot kill the scheduler |
| T-02-14 | Repudiation | job run history | medium | mitigate | JobRun row per execution (status/rows/timestamps); failures also notify System Admins for real-time surfacing |
| T-02-SC | Tampering | APScheduler dependency (pip) | high | mitigate | No NEW package (APScheduler already vetted/present since a prior phase); pin tightened to >=3.10,<4 to block the unstable 4.0 pre-release rewrite. No install of a new/unaudited package -> no legitimacy checkpoint needed |
</threat_model>
