# Phase 03.1 Plan 01: Install & Wire python-social-auth (Entra ID PKCE backend) Summary

Wired `social-auth-app-django==6.0.0` into the running Django app: a PKCE-honoring
`AzureADTenantOAuth2PKCE` backend is registered first in `AUTHENTICATION_BACKENDS`,
the full `SOCIAL_AUTH_*` settings block (pinned redirect, refusal-safe middleware,
customized create-user-free pipeline) is in place, the `auth/` URL include resolves
the registered Entra callback, and all 17 `social_django` migrations applied cleanly
on MSSQL LocalDB — the structural foundation the real SSO round-trip stands on.

## What Was Built

**Task 1 — Dependency pin + regression guard** (`b896ab8`)
- Added `social-auth-app-django==6.0.0` to `requirements.txt` (exact pin, D-01).
- Replaced the stale "not yet wired / cutover phase" comment on
  PyJWT/cryptography/requests with a phase-agnostic note that they are now
  transitive deps of `social-auth-core` (Pitfall 8).
- `pip install` pulled `social-auth-core==5.0.2` + `oauthlib`, `requests-oauthlib`,
  `defusedxml`, `python3-openid` — no compiler prompt (Windows wheels).
  cryptography/PyJWT/requests were already satisfied at the required minimums.
- `manage.py check` clean; existing `accounts web` suite green post-upgrade.

**Task 2 — PKCE backend + settings/urls wiring + migrate** (`20fc47a`)
- `accounts/backends.py`: `AzureADTenantOAuth2PKCE(BaseOAuth2PKCE, AzureADTenantOAuth2)`
  with `DEFAULT_USE_PKCE = True` and **no** `name` override — so the callback URL and
  every `SOCIAL_AUTH_AZUREAD_TENANT_OAUTH2_*` prefix stay `azuread-tenant-oauth2`
  (Pitfall 1: the stock backend silently ignores `USE_PKCE`).
- `config/settings.py`:
  - `social_django` added to `INSTALLED_APPS`.
  - `social_django.middleware.SocialAuthExceptionMiddleware` inserted **after**
    `AuthenticationMiddleware` and **before** `MessageMiddleware`.
  - Two social context processors (`backends`, `login_redirect`) added.
  - `AUTHENTICATION_BACKENDS`: PKCE subclass first, `ModelBackend` second (D-01/D-03).
  - `SOCIAL_AUTH_*` block: KEY/SECRET/TENANT_ID from `.env` (D-04), `USE_PKCE=True`
    (D-02), pinned `REDIRECT_URI` (Pitfall 3), `RAISE_EXCEPTIONS=False` +
    `LOGIN_ERROR_URL="/login"` (D-06/D-09#2), and the customized `SOCIAL_AUTH_PIPELINE`
    (`associate_by_email` -> `deny_unprovisioned` -> `associate_user` ->
    `write_azure_oid`, `create_user` removed).
  - `REST_FRAMEWORK` left untouched (D-10).
- `config/urls.py`: `path("auth/", include("social_django.urls", namespace="social"))`.
- `manage.py migrate social_django`: all 17 migrations applied on MSSQL LocalDB;
  the 5 `social_auth_*` tables (`usersocialauth`, `nonce`, `association`, `code`,
  `partial`) exist — resolves research assumption A2 (no fix-forward RunSQL needed).

## Known Stubs

The `SOCIAL_AUTH_PIPELINE` references `accounts.pipeline.deny_unprovisioned` and
`accounts.pipeline.write_azure_oid`, which do **not** exist yet — **Plan 03.1-02**
creates them. This is an intentional, plan-sanctioned forward dependency: the
pipeline is a lazy dotted-string tuple resolved only during a live auth request, so
`check` / `migrate` / the existing test suite never import it and all pass. It does
**not** block this plan's goal (structural wiring); it is verified resolved when
Plan 02 lands `accounts/pipeline.py` and Plan 04 exercises the deny path.
