# Phase 03.1: Authentication — Entra ID SSO (local-dev proof) - Context

**Gathered:** 2026-07-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Wire real Microsoft Entra ID sign-in into the running Django app on `localhost`,
making it the **real** authentication path that replaces the DEBUG dev-login stub —
proven end-to-end with a real MMCM tenant account. This is a **local-dev proof**:
the goal is to demonstrate the full SSO round-trip works against the project-owned
Entra tenant, not to ship production auth. Production cutover (https redirect,
secret rotation, staging verification) stays in the deploy phase.

**In scope:** OAuth Authorization-Code + PKCE login via `social-auth-app-django`'s
`AzureADTenantOAuth2` backend; linking an Entra identity to a pre-seeded role-user;
refusing unprovisioned accounts; keeping the dev-login stub behind `DEBUG`;
session-based auth; local logout.

**Out of scope (deferred to deploy phase):** production https redirect URI,
client-secret rotation, staging verification, Entra global sign-out, backend-issued
JWT / API tokens, group/app-role-based provisioning.
</domain>

<decisions>
## Implementation Decisions

### Authentication Backend (LOCKED — carried from 2026-07-03 user decision)
- **D-01:** Use `social-auth-app-django`'s `AzureADTenantOAuth2` backend
  (`social_core.backends.azuread_tenant.AzureADTenantOAuth2`) — **NOT** a custom
  MSAL/PKCE flow. Rationale: the Azure app registration's redirect URI is already
  set to `http://localhost:8000/auth/complete/azuread-tenant-oauth2/`, which is
  exactly python-social-auth's callback convention. Conform the app to the
  registered credentials, not the reverse.
- **D-02:** Enable PKCE: `SOCIAL_AUTH_AZUREAD_TENANT_OAUTH2_USE_PKCE = True`. This
  satisfies the ROADMAP/SRS "Authorization Code + PKCE" wording — social-core
  supports PKCE on this backend, so there is no conflict, just make it explicit.
- **D-03:** Keep `django.contrib.auth.backends.ModelBackend` in
  `AUTHENTICATION_BACKENDS` alongside the Entra backend so the break-glass
  superuser can still log in with a password if `DEBUG=False`.
- **D-04:** Creds are read from `.env` (already present): `SOCIAL_AUTH_AZUREAD_TENANT_OAUTH2_KEY`,
  `SOCIAL_AUTH_AZUREAD_TENANT_OAUTH2_SECRET`, `SOCIAL_AUTH_AZUREAD_TENANT_OAUTH2_TENANT_ID`.
  The client-credentials flow against the tenant was already verified working
  (tenant id / client id / secret are correct).

### Identity → Seeded-User Linking
- **D-05:** On first Entra login, match to an **existing** seeded `User` by
  **email/UPN** (social-auth `associate_by_email`), then write `User.azure_oid`.
  Subsequent logins match by `azure_oid` (the durable key); email is only the
  first-login bridge. Safe because `AzureADTenantOAuth2` is single-tenant, so the
  matched email is tenant-verified.
- **D-06:** **Unprovisioned = refused.** If no seeded `User` matches, the login is
  refused — **no auto-creation** of users. Implement by customizing
  `SOCIAL_AUTH_PIPELINE` to remove `create_user` and insert a deny/short-circuit
  step for the no-match case.
- **D-07:** Provide a one-off management command
  `manage.py link_entra <username> <real-UPN>` that repoints a seeded user's email
  to the tester's real MMCM UPN so email-match can link. Reversible, re-runnable,
  and does **not** mutate `seed_demo.py` (keeps the committed demo seed synthetic).
  For this proof, bind the **faculty** slot (`username="mayo"`).

### Dev-Login Coexistence
- **D-08:** Keep **both** login paths. "Sign in with Microsoft" is the real auth
  path (works on localhost too). The username dev-login list + its POST handler
  stay gated behind `settings.DEBUG` (the existing `web/views.py` stub already
  gates on `DEBUG`). Under `DEBUG=False` the page shows the Entra button only.

### Proof Scope / Success Criteria
- **D-09:** "Proven" requires all three:
  1. A real Microsoft login round-trips and lands on the **correct role-home**
     (faculty) for the bound account.
  2. An authenticated tenant account with **no provisioned User is refused**
     (not created, not logged in).
  3. The dev-login stub still works under `DEBUG`, and the superuser break-glass
     still logs in.

### Session & Logout
- **D-10:** **Session-only.** social-auth establishes a standard Django session;
  DRF stays on `SessionAuthentication` + `IsAuthenticated`. The session-vs-JWT
  divergence noted in `docs/USE_CASES.md` is **explicitly deferred** — not resolved
  this phase.
- **D-11:** Logout = local Django session flush → redirect to login. **No** Entra
  global sign-out this phase.

### Claude's Discretion
- Exact pipeline function ordering and the shape of the deny-if-unprovisioned step.
- Whether a successful/refused login writes an `AuditLog` row (project convention
  favors auditing write/security events — planner may add it; not required for the
  proof).
- Login template markup/placement of the Microsoft button (keep consistent with
  the existing Franken UI login page).
- Optionally updating the bound faculty user's display name to the tester's real
  name (cosmetic; the seed name "Jane Mayo" otherwise remains).

### Folded Todos
- **`entra-auth-backend-decision.md`** (was `phase_target: 8`) — folded into this
  phase because Entra was reprioritized ahead of Phase 4 as a local-dev proof
  (see ROADMAP evolution / STATE.md). Supplies D-01…D-06 and these watch-outs the
  planner must carry:
  - Reconcile the ROADMAP/SRS "Authorization Code + PKCE" wording by enabling PKCE
    on the backend (D-02) — make it explicit in the plan.
  - `.env.example` / `requirements.txt` comments say Entra is "Phase 2" — **stale**;
    fix them when auth lands.
  - The current dev client secret was shared in a chat transcript — **rotate before
    prod** (deploy phase, not this phase).
  - Add the production https redirect URI to the Azure app registration before
    cutover (deploy phase).
