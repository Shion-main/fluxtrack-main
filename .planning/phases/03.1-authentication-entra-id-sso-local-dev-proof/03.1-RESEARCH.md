## Summary

This phase wires `social-auth-app-django` (social_django) into an existing Django 6.0.6 app so the
`AzureADTenantOAuth2` backend performs the OAuth2 Authorization-Code login round-trip against the
project-owned MMCM Entra tenant on `http://localhost:8000`. All the hard parts are *wiring*, not
algorithm design: settings keys, one URL include, a customized `SOCIAL_AUTH_PIPELINE`, a trivial
management command, and running the library's migrations on MSSQL LocalDB. The backend, PKCE
decision, env-var names, and callback URL are already locked and match the registered Azure app
registration exactly — the job is to conform the code to those credentials.

Three findings materially change the plan and were verified by reading the pinned source
(`social-auth-core 5.0.2`, `social-auth-app-django 6.0.0`): **(1)** the stock `AzureADTenantOAuth2`
does **not** inherit the PKCE mixin, so `USE_PKCE = True` is silently ignored — genuinely honoring
D-02 requires a 3-line subclass mixing in `BaseOAuth2PKCE`; **(2)** adding a second entry to
`AUTHENTICATION_BACKENDS` will **break the existing dev-login stub** (`login(request, user)` raises
`ValueError` with multiple backends) unless the call passes `backend="...ModelBackend"`; **(3)** the
backend's social uid is the `sub` claim, not `oid` — so `User.azure_oid` must be written from
`response['oid']` in a custom pipeline step, not from the social uid.

**Primary recommendation:** Pin `social-auth-app-django==6.0.0`; add a tiny PKCE-mixin subclass of
`AzureADTenantOAuth2` (keeps `name="azuread-tenant-oauth2"`, so the callback URL and all
`SOCIAL_AUTH_AZUREAD_TENANT_OAUTH2_*` env keys are unchanged); wire `social_django.urls` under
`auth/`; customize the pipeline to `associate_by_email → write azure_oid from oid → deny-if-no-user
(raise AuthForbidden)` with `create_user` removed; fix the dev-login `login()` call to name the
ModelBackend; run `migrate social_django` on MSSQL and verify. Validate wiring and the deny step with
Django `TestCase`; the interactive Microsoft round-trip (D-09 #1) is a manual browser step.

### Pattern 3: Custom pipeline steps
```python
# accounts/pipeline.py
from social_core.exceptions import AuthForbidden

def deny_unprovisioned(strategy, backend, details, user=None, *args, **kwargs):
    """D-06/AUTH-03: refuse any tenant identity with no pre-provisioned, active User.
    Runs after associate_by_email; create_user is removed from the pipeline."""
    if user is None or not user.is_active:      # is_active covers AUTH-05
        raise AuthForbidden(backend)            # SocialAuthExceptionMiddleware -> LOGIN_ERROR_URL

def write_azure_oid(strategy, backend, details, response, user=None, *args, **kwargs):
    """D-05: persist the durable Entra object id. The backend's social uid is the
    'sub' claim, NOT 'oid' — so read response['oid'] explicitly."""
    if user is None:
        return
    oid = response.get("oid")
    if oid and user.azure_oid != oid:
        user.azure_oid = oid
        user.save(update_fields=["azure_oid"])
    # planner's discretion: emit AuditLog(actor=user, event_type="auth.entra_login")
```

### Anti-Patterns to Avoid
- **Setting `USE_PKCE=True` on the stock backend and assuming PKCE happens.** It does not (Pitfall 1).
- **Calling `django.contrib.auth.login(request, user)` without a `backend=` arg** once two backends
  are configured — raises `ValueError` (Pitfall 2).
- **Writing `User.azure_oid` from the social uid.** The uid is `sub` (pairwise, per-app), not `oid`.
- **Leaving `create_user` in the pipeline** — it would auto-provision unprovisioned tenant accounts,
  violating D-06/AUTH-03.
- **Accessing the app via `http://127.0.0.1:8000`** during the proof — the derived redirect_uri
  would be `127.0.0.1`, not the registered `localhost` (Pitfall 3).

## Common Pitfalls

### Pitfall 1: `USE_PKCE=True` is silently ignored by the stock backend (breaks the D-02 / AUTH-01 promise)
**What goes wrong:** `AzureADTenantOAuth2(AzureADOAuth2 → BaseOAuth2)` does **not** inherit
`BaseOAuth2PKCE`. The PKCE logic (`code_challenge`, `code_verifier`) lives only in
`BaseOAuth2PKCE.auth_params` / `.auth_complete_params`. `BaseOAuth2.auth_params` has no PKCE branch,
so `SOCIAL_AUTH_..._USE_PKCE = True` sets a value nothing reads — the flow runs as plain
Authorization Code (which *works* because this is a confidential client with a secret, so Azure does
not require PKCE — masking the omission).
**Why it happens:** class hierarchy — verified in `social_core/backends/azuread_tenant.py` and
`oauth.py` (PKCE grep hits only inside `class BaseOAuth2PKCE`).
**How to avoid:** define `AzureADTenantOAuth2PKCE(BaseOAuth2PKCE, AzureADTenantOAuth2)` (Pattern 2)
and reference it in `AUTHENTICATION_BACKENDS`. Name/prefix/callback URL are all inherited unchanged.
**Warning signs:** the authorize-redirect URL in the browser has no `code_challenge` query param.
[VERIFIED: sdist source, social-auth-core 5.0.2]

### Pitfall 2: Adding a second backend breaks the existing dev-login stub
**What goes wrong:** `web/views.py:63` calls `login(request, user)` with no backend arg. Today
`AUTHENTICATION_BACKENDS` is unset → Django defaults to a single `ModelBackend`, so this works. Once
this phase sets a 2-entry `AUTHENTICATION_BACKENDS`, Django's `login()` raises
`ValueError: You have multiple authentication backends configured and therefore must provide the
backend argument...`. This regresses D-08 / D-09 #3.
**Why it happens:** `django.contrib.auth.login` needs `user.backend` or an explicit `backend=` when
>1 backend is configured and the user wasn't produced by `authenticate()`.
**How to avoid:** change the stub to
`login(request, user, backend="django.contrib.auth.backends.ModelBackend")`.
**Warning signs:** dev-login POST 500s with ValueError after settings change.
[VERIFIED: codebase `web/views.py:57-64` + Django auth semantics]

### Pitfall 3: Redirect-URI exact-match (localhost vs 127.0.0.1, http, trailing slash)
**What goes wrong:** Entra requires the `redirect_uri` to *exactly* match a registered value. social-auth
derives it from the request host unless pinned. Visiting `http://127.0.0.1:8000` yields a
`127.0.0.1` redirect_uri that is not registered → `AADSTS50011 redirect URI mismatch`.
**How to avoid:** (a) always open `http://localhost:8000`; (b) pin
`SOCIAL_AUTH_AZUREAD_TENANT_OAUTH2_REDIRECT_URI` to the exact registered string (Pattern 1);
(c) trailing slash matters — `social_django.urls` uses `TRAILING_SLASH=True` by default, giving
`/auth/complete/azuread-tenant-oauth2/` which matches the registration. [VERIFIED: `social_django/urls.py`]

### Pitfall 4: http-on-localhost / secure-cookie / HTTPS forcing
**What goes wrong:** if `SOCIAL_AUTH_REDIRECT_IS_HTTPS`, `SESSION_COOKIE_SECURE`, or
`SECURE_SSL_REDIRECT` are enabled, the http-localhost flow breaks (verifier/session lost or redirect
upgraded to https).
**How to avoid:** keep all three at their defaults (False) for the local proof; DEBUG=True is fine.
These get enabled only in the deferred deploy phase.
**Warning signs:** session/state lost between begin and complete; browser blocked on https upgrade.

### Pitfall 5: `SOCIAL_AUTH_JSONFIELD_ENABLED` is a no-op in social_django 6.0.0
**What goes wrong:** older guidance says you must set `SOCIAL_AUTH_JSONFIELD_ENABLED = True` on
non-Postgres DBs or `extra_data` stores as text. In 6.0.0, `UserSocialAuth.extra_data` and
`Partial.data` are **unconditionally** `models.JSONField` (migrations 0012–0016 finalize this), so the
setting no longer changes the schema.
**Why it happens:** the historical toggle in `social_django/fields.py` is legacy; the model uses
native `models.JSONField` directly.
**How to avoid:** you may set it for clarity but it is not required; on MSSQL `JSONField` maps to
`nvarchar(max)` with JSON validation via mssql-django 1.7.3. Don't block the plan on it.
[VERIFIED: `social_django/models.py:34,127` + migrations list]

### Pitfall 6: Dependency upgrade surprise (cryptography 42→46, PyJWT, requests)
**What goes wrong:** social-auth-core 5.0.2 requires `cryptography>=46.0.7`, `PyJWT[crypto]>=2.13.0`,
`requests>=2.34.0`. The current `>=` pins are satisfied by the higher versions, so pip silently
upgrades them and adds `oauthlib`, `requests-oauthlib`, `defusedxml`, `python3-openid`.
**How to avoid:** run the install in a controlled step, then run the existing test suite
(`manage.py test`) to confirm no regression from the cryptography/requests bump. cryptography 46 has
prebuilt Windows wheels (no compiler needed). Optionally bump the requirements.txt pins to match the
new minimums for reproducibility. [VERIFIED: sdist PKG-INFO]

### Pitfall 7: social_django migrations on MSSQL LocalDB
**What goes wrong:** `migrate` applies 17 social_django migrations, including a data migration
(0013 copies extra_data) and multiple ALTER/rename steps (0012–0016) plus a `CheckConstraint`
(0017, `uid != ''`). Phase 1 already hit mssql-django quirks (AlterField no-op SQL on `db_collation`),
so there is precedent for MSSQL edge behavior.
**Why it happens:** on a *fresh* install the 0012–0016 sequence still runs step-by-step; mssql-django
translates ALTER COLUMN + sp_rename.
**How to avoid:** run `python manage.py migrate social_django` and confirm all 5 tables
(`social_auth_usersocialauth`, `_nonce`, `_association`, `_code`, `_partial`) are created; inspect with
`sqlmigrate social_django 0017` if anything errors. `unique_together (provider, uid)` = unique index on
`varchar(32)+nvarchar(255)` (≈542 bytes, under the 900-byte limit) — safe. LOW risk but verify.
[VERIFIED: migrations list + models; MEDIUM confidence on MSSQL runtime]

### Pitfall 8: Stale "Phase 2" / "Phase 8" comments

### link_entra management command (ASCII-only output per convention #4)
```python
# accounts/management/commands/link_entra.py
from django.contrib.auth import get_user_model
from django.core.exceptions import ValidationError
from django.core.management.base import BaseCommand, CommandError
from django.core.validators import validate_email

class Command(BaseCommand):
    help = "Repoint a seeded user's email to a real MMCM UPN for Entra email-match linking."

    def add_arguments(self, parser):
        parser.add_argument("username")
        parser.add_argument("upn")

    def handle(self, *args, **o):
        User = get_user_model()
        try:
            validate_email(o["upn"])
        except ValidationError:
            raise CommandError(f"Invalid UPN/email: {o['upn']}")
        try:
            u = User.objects.get(username=o["username"])
        except User.DoesNotExist:
            raise CommandError(f"No user with username '{o['username']}'")
        clash = User.objects.filter(email__iexact=o["upn"]).exclude(pk=u.pk)
        if clash.exists():
            raise CommandError(f"UPN already used by: {clash.first().username}")
        if u.email == o["upn"]:
            self.stdout.write(self.style.SUCCESS(f"No change -> {u.username} already {o['upn']}"))
            return
        u.email = o["upn"]
        u.save(update_fields=["email"])
        self.stdout.write(self.style.SUCCESS(f"Linked {u.username} -> {o['upn']}"))
```
Idempotency: re-running with the same UPN prints "No change" and exits 0. Validation: rejects bad
email and cross-user clashes (which would make `associate_by_email` raise on >1 match). ASCII `->`
arrows only.

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | OAuth2 Auth-Code (+PKCE via mixin) delegated to Entra; social-core verifies id_token RS256 sig + issuer + `tid` |

### Known Threat Patterns for Django + Entra OAuth2
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Account takeover via unverified email match | Spoofing | Single-tenant backend enforces `tid`; email is org-directory-verified (D-05); `associate_by_email` raises on >1 match |
| Auto-provisioning of any tenant user | Elevation | `create_user` removed + `deny_unprovisioned` (D-06/AUTH-03) |
| Authorization-code interception | Tampering | PKCE (Pattern 2) + `state` CSRF param |
| Redirect-URI manipulation | Tampering | Exact-match registered URI, pinned `REDIRECT_URI` (Pitfall 3) |
| CSRF on callback | Spoofing | social-auth `state` param; `complete` view is `@csrf_exempt` by design (GET callback) [VERIFIED] |
| Secret leakage | Info disclosure | Dev secret was shared in a transcript — **rotate before prod** (deferred to deploy phase, per watch-out) |
| Reactivation bypass | Elevation | `deny_unprovisioned` also checks `is_active` (AUTH-05) |
