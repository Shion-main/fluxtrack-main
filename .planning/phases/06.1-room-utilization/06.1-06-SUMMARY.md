# Phase 06.1 Plan 06: The /ifo/utilization Page Summary

T2 has a surface. The heat grid, the building and floor rollup, the least-used
rooms table and the saturated-blocks ranking sit on one IFO console page,
reachable from the nav and from the dashboard's Room Occupancy card.

## What shipped

`web.ifo.utilization` at `/ifo/utilization` (name `ifo_utilization`), with FOUR
independent `safe_card` owners -- `room_heat_grid`, `building_floor_rollup`,
`room_breakdown`, `block_saturation` -- plus `room_utilization` for the summary
sentence. Each is guarded separately in the template. `paginate(request,
breakdown[0] or [])` carries the plan-03 `or []` guard forward; without it a
`room_breakdown` failure would reach `Paginator(None)` and 500 the page,
reintroducing the exact bug plan 03 fixed on the dashboard.

Nav entry under Reporting between Dashboard and Weekly reports, hardcoded path
with the url-name condition repeated for `is-active` and `aria-current`, matching
the local convention. The dashboard's occupancy figure is now a link forwarding
`?from=&to=`, the same shape as the faculty-to-scorecard link below it.

## The heat ramp

Five steps over the existing `--navy-tint` family, added to `static/css/timetable.css`
so ONE file governs both grids. Measured contrast, recorded in a comment above the
ramp:

| step | background | foreground | ratio |
|------|-----------|------------|-------|
| h0 | `#f7f9fd` | `--ink #0b1b3a` | 16.17:1 |
| h1 | `#e7ecf7` | `--ink #0b1b3a` | 14.40:1 |
| h2 | `#cfdaf0` | `--ink #0b1b3a` | 12.13:1 |
| h3 | `#a7bce4` | `--ink #0b1b3a` | 8.89:1 |
| h4 | `#2f4b86` | `#ffffff` | 8.50:1 |

Every step clears WCAG AA (4.5:1) with room to spare. `h4` takes a light
foreground rather than a lightened background, so the top of the ramp stays
visibly the top.

**Colour is never alone.** Every cell prints its percentage AND its used hours as
visible text. A cell the term does not timetable renders in the established
free-cell idiom -- a dash marked `aria-hidden` beside an `sr-only` "not
timetabled" -- never a misleading 0%. Under print the ramp flattens to white with
true-black borders and step 4's foreground returns to black, keeping the numbers,
which were always the information; five tints of one hue come off a mono office
printer as two indistinguishable greys.

## The scale is relative, and it says so

Consistent with the prior executor's deliberate choice to make the occupancy pill
neutral: **no threshold policy was invented.** Shading is relative to the range
being rendered -- palest is least-used, darkest is most-used, within these dates.
The table caption and the on-screen legend both state that in words, so a reader
knows the colour is a rank and not a grade.

**One implementation reversal worth recording.** The first ramp scaled each cell
against the busiest cell (`ceil(pct * 4 / max)`). On live data that bunched badly:
39 of 67 cells landed on the top step, because campus utilization clusters in a
narrow band whose floor is nowhere near zero, so almost every cell sits within
75% of the maximum. Replaced with **equal-count rank buckets** over the used
cells, with ties sharing a step. Live distribution is now 2 / 16 / 18 / 14 / 22,
which spreads across the distribution that actually exists -- the whole job of a
relative scale.

error text, that the raw exception never reaches the page, and that the other
sections still rendered. `assertLogs` both confirms `safe_card` logged
server-side and keeps the deliberate traceback out of the runner output.
