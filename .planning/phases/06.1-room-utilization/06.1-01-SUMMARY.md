# Phase 06.1 Plan 01: Campus Block Ladder + T1 Room-Hours Summary

One derivation of the campus block ladder shared by the printed timetable and the
new `room_utilization` aggregate, which reports booked vs used vs available
room-hours with waste derived from actual timestamps rather than the `ended_early`
flag.

## What Was Built

**Task 1 — the shared ladder** (`899e444`). `CampusBlock` (frozen dataclass),
`campus_block_ladder(term)` and `teaching_weekdays(term)` (the latter since
replaced by `timetabled_cells`, see the D-10 amendment below) in
`scheduling/reporting.py`, plus the pure helpers `_span_seconds` and
`_block_duration_seconds`. `web/room_state.room_timetable` now delegates its
inline `sorted(set(...))` derivation to the shared function and takes
`slots = [b.start for b in blocks]`; its return shape, half-open occupancy rule
and capacity computation are untouched. The now-orphaned bare `Schedule` import
was removed from `room_state.py` (`ScheduleStatus` is still used).

**Task 2 — the T1 metric** (`f655351`). `RoomUtilization` dataclass,
`room_utilization(*, start, end, term, as_of=None)`, `_session_contribution`,
`_physical_rooms`, `_exclude_virtual`, `_hours_pct`, `_hours`.

**Task 3 — DB-free arithmetic tests** (`bc7525b`). `scheduling/tests_reporting_rooms.py`,
25 `SimpleTestCase` tests across four classes. Plan 02 extends this same module
with DB-backed fixture tests.

## Chosen Block-Duration Rule

The MODAL span of the schedules starting at that block wins, so one mis-imported
outlier can neither stretch nor shrink a rung. A frequency tie resolves to the
LONGEST span — an understated block understates `available_hours` and would
silently INFLATE the utilization rate, which is the failure direction that
matters.

## Observations Against the Seeded Term (`2nd Term SY 2025-2026`)

**Ladder length: 11 blocks.** 07:00 through 19:30, every rung exactly 75 minutes,
starts spaced exactly 75 minutes apart. Summed block time = 825 min/day = 13.75 h.

**LADDER-TILING ANSWER: the ladder tiles cleanly. Zero overlaps.** For every
adjacent pair, `block.start + block.duration == next_block.start` exactly
(07:00+75m = 08:15, and so on up the ladder). No block's end exceeds the next
block's start, so summing block durations does NOT overstate wall-clock capacity
and the `available_hours` denominator is sound as built. No revisiting needed.

**Timetabled cells: 67 of a possible 77.** Mon-Sat carry all 11 rungs each;
Sunday carries exactly 1. Sunday IS a teaching day — it simply is not a full one,
which is the whole content of the D-10 amendment below.

**Physical rooms: 125** (V-prefixed rooms excluded).

**Current week (2026-07-13 .. 2026-07-19, `as_of` 2026-07-19).** The AFTER column
is the shipped D-10 cell-sum denominator; BEFORE is the superseded `days x blocks`
cross-product, kept so the two are directly comparable:

confirms `seed_term`'s pre-`scheduled_start` check-ins; without the clamp those

## Plan Assumptions That Turned Out Wrong

**The seeded term contains a Sunday schedule, so `teaching_days` is 7, not 6.**
The plan's verification step expected "weekdays including 5 (Saturday) but
excluding 6 (Sunday)" and instructed: "If Sunday appears, the term contains Sunday
schedules and the derivation is right — investigate the data, not the code."
Investigated: `Schedule.objects.filter(term=t, status=ACTIVE, day_of_week=6).count()`
returns **1**. A single Sunday schedule row.

This is D-06 behaving exactly as designed — the derivation is correct — but it has
an outsized denominator effect worth flagging to plan 02 and to the owner: that one
row adds a full seventh teaching day, i.e. `125 rooms x 13.75 h = 1718.75 h` to a
12031.3 h denominator. Utilization reads 23%; excluding Sunday it would read
roughly 27%. **The number is honestly derived, but it is sensitive to a single data
row.** If that lone Sunday schedule is an import artifact rather than a real class,
the metric will move ~4 points once it is corrected. Recommendation: the owner
should confirm whether MMCM genuinely timetables that Sunday slot. No code change
is warranted either way — hardcoding a day exclusion is precisely what D-06
forbids.

**RESOLVED 2026-07-19 by D-10** — see the amendment section below. The owner's
answer was neither "exclude Sunday" nor "leave it": the finding was promoted to a
decision that changes the denominator's SHAPE, so the sensitivity disappears
without anyone ruling on whether that one row is genuine.

## D-10 Amendment — the denominator is a (day, block) cell sum

**Shipped in `7521339`, after the four Wave 1 commits.** Denominator-only:
`_session_contribution`, the streamed fold and every numerator figure are
untouched and unchanged.

**The change.** From a cross-product to a sum over the cells really timetabled:

```
BEFORE:  available = physical_rooms x teaching_days x SUM(block durations)
AFTER:   available = SUM over timetabled (day, block) cells of
                     (physical_rooms x block_duration)
```

**Why.** The Wave 1 finding above, promoted to a locked decision. One Sunday row
against 289 Saturday ones must not move the headline metric by four points. D-10
applies D-01's own rationale one level finer: a block nobody schedules is not
waste, and neither is a (day, block) cell nobody schedules.

**API changes plans 03-07 must know about:**

| Change | Detail |
|---|---|
| **REMOVED** `teaching_weekdays(term)` | Replaced, not kept. It would have become dead code, and a helper that returns a day list is actively misleading once days are no longer the unit of capacity. No caller outside `reporting.py` existed. |
| **NEW** `timetabled_cells(term)` | Returns the ordered `(day, block)` cell SET as `TimetabledCell(day, start, duration_seconds)`, `[]` for a None/empty term. Shaped for plan 04's heat grid — it wants exactly "which cells exist and how long is each", not a day list. |
| **NEW** `TimetabledCell` frozen dataclass | `day` is a `DayOfWeek` value (MON=0..SUN=6, directly comparable to `date.weekday()`); `duration_seconds` is the ladder rung's own duration, so a cell and the printed grid cannot disagree. |
| **NEW field** `RoomUtilization.timetabled_cells: int` | Cells summed across the calendar range. Purely additive. |
| **UNCHANGED fields** `physical_rooms`, `blocks_per_day`, `teaching_days` | All three still render. `blocks_per_day` remains `len(campus_block_ladder)` — the CAMPUS ladder height, i.e. the grid's height, which was always its meaning; it is no longer a per-day block count, because under D-10 that varies by day. `teaching_days` keeps an honest meaning: calendar days in range carrying at least one timetabled cell. |

**A cell is claimed by the block a schedule STARTS in**, keyed on distinct
`(day_of_week, start_time)` pairs — the same keying the ladder rungs themselves
are derived from. A long class therefore claims one cell rather than every rung
its window crosses; the campus-wide union across all 125 rooms is what makes a
cell a real teaching slot. This is why the 15:45-20:45 Sunday class yields
Sunday 1 cell, not 4.

**Still fully derived.** No threshold, no magic number, no hardcoded day list.
The literal scan is re-run in Verification Results below and is clean.

**`DenominatorTests` (new, 9 DB-backed tests).** The independent-recomputation
pattern the owner asked to be preserved is ported to the cell-sum form rather
than deleted: `_expected_available_seconds` rebuilds the denominator straight
from the `Schedule` rows in plain Python, calling none of the helpers the
implementation uses. The fixture reproduces the D-10 shape in miniature — two
dense days on a deliberately 3-rung ladder (never the live 11) plus one outlier
day carrying a single class. `test_cross_product_would_overstate_and_is_not_what_ships`
pins the GAP between the cell count and `teaching_days x blocks_per_day`, so a
silent revert to the cross-product cannot pass.
