# Phase 06.1 Plan 01: Campus Block Ladder + T1 Room-Hours Summary

One derivation of the campus block ladder shared by the printed timetable and the
new `room_utilization` aggregate, which reports booked vs used vs available
room-hours with waste derived from actual timestamps rather than the `ended_early`
flag.

## What Was Built

**Task 1 — the shared ladder** (`899e444`). `CampusBlock` (frozen dataclass),
`campus_block_ladder(term)` and `teaching_weekdays(term)` in
`scheduling/reporting.py`, plus the pure helpers `_span_seconds` and
`_block_duration_seconds`. `web/room_state.room_timetable` now delegates its
inline `sorted(set(...))` derivation to the shared function and takes
`slots = [b.start for b in blocks]`; its return shape, half-open occupancy rule
and capacity computation are untouched. The now-orphaned bare `Schedule` import
was removed from `room_state.py` (`ScheduleStatus` is still used).

**Task 2 — the T1 metric** (`f655351`). `RoomUtilization` dataclass,
`room_utilization(*, start, end, term, as_of=None)`, `_session_contribution`,
`_physical_rooms`, `_exclude_virtual`, `_hours_pct`, `_hours`.

**Task 3 — DB-free arithmetic tests** (`bc7525b`). `scheduling/tests_reporting_rooms.py`,
25 `SimpleTestCase` tests across four classes. Plan 02 extends this same module
with DB-backed fixture tests.

## Chosen Block-Duration Rule

The MODAL span of the schedules starting at that block wins, so one mis-imported
outlier can neither stretch nor shrink a rung. A frequency tie resolves to the
LONGEST span — an understated block understates `available_hours` and would
silently INFLATE the utilization rate, which is the failure direction that
matters.

## Observations Against the Seeded Term (`2nd Term SY 2025-2026`)

**Ladder length: 11 blocks.** 07:00 through 19:30, every rung exactly 75 minutes,
starts spaced exactly 75 minutes apart. Summed block time = 825 min/day = 13.75 h.

**LADDER-TILING ANSWER: the ladder tiles cleanly. Zero overlaps.** For every
adjacent pair, `block.start + block.duration == next_block.start` exactly
(07:00+75m = 08:15, and so on up the ladder). No block's end exceeds the next
block's start, so summing block durations does NOT overstate wall-clock capacity
and the `available_hours` denominator is sound as built. No revisiting needed.

**Teaching weekdays: `[0, 1, 2, 3, 4, 5, 6]` — Sunday IS included.** See the
plan-assumption section below; this is a data finding, not a code defect.

**Physical rooms: 125** (V-prefixed rooms excluded).

## Plan Assumptions That Turned Out Wrong

**The seeded term contains a Sunday schedule, so `teaching_days` is 7, not 6.**
The plan's verification step expected "weekdays including 5 (Saturday) but
excluding 6 (Sunday)" and instructed: "If Sunday appears, the term contains Sunday
schedules and the derivation is right — investigate the data, not the code."
Investigated: `Schedule.objects.filter(term=t, status=ACTIVE, day_of_week=6).count()`
returns **1**. A single Sunday schedule row.

This is D-06 behaving exactly as designed — the derivation is correct — but it has
an outsized denominator effect worth flagging to plan 02 and to the owner: that one
row adds a full seventh teaching day, i.e. `125 rooms x 13.75 h = 1718.75 h` to a
12031.3 h denominator. Utilization reads 23%; excluding Sunday it would read
roughly 27%. **The number is honestly derived, but it is sensitive to a single data
row.** If that lone Sunday schedule is an import artifact rather than a real class,
the metric will move ~4 points once it is corrected. Recommendation: the owner
should confirm whether MMCM genuinely timetables that Sunday slot. No code change
is warranted either way — hardcoding a day exclusion is precisely what D-06
forbids.
