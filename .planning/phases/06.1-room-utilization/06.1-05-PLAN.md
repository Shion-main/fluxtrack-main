<traps>
- **A zero-session room must not vanish.** If the per-room aggregate is built by
  grouping the SESSION queryset, a room with no sessions produces no group and
  drops out entirely -- and it is precisely the least-used room, the single most
  useful row in the table. Start from the ROOM universe and left-join the session
  totals onto it, or group the sessions and then fill in the missing rooms from
  `_physical_rooms()`.
- **Do not merge the room and floor levels into one multi-join query.** The
  module's separate-query rule (`reporting.py:129-141`) exists because reverse-join
  row multiplication inflates counts. Derive rooms once and reduce upward in
  Python.
- **`Floor` uniqueness is per building.** `unique_together = [("building", "number")]`
  (`campus/models.py:18-19`), so floor 3 exists in every building. Key floor rows
  by the building-and-number pair, never by number alone, or two buildings' third
  floors merge into one row.
- **MSSQL.** No unbounded `pk__in`. Materialize with `list()` before follow-up
  queries inside an `.iterator()`.
</traps>

  <action>
Add to `scheduling/reporting.py`, after `block_saturation`.

Define a `RoomLoad` dataclass: `room_id`, `code`, `name`, `building_code`,
`building_name`, `floor_number`, `capacity`, `used_hours`, `booked_hours`,
`available_hours`, `wasted_hours`, `utilization_pct`, `session_count`,

Define `room_breakdown(*, start, end, term, as_of=None)`, keyword-only.

Implementation:

1. Start from the room universe, not the sessions:
   `_physical_rooms().select_related("floor__building")`, ordered by building code
   then floor number then room code. Materialize it with `list()`. This is what
   guarantees a never-used room still gets a row, and it is bounded by the campus
   room count so materializing is safe.

2. Compute the shared per-room available seconds once -- blocks per day from
   `campus_block_ladder`, teaching days in range from `teaching_weekdays`, exactly
   as `room_utilization` does. Every physical room has the same denominator under
   D-01, so this is one figure reused, not a per-room query.

3. Group the session side in a single pass:
   `_exclude_virtual(_scoped_sessions(...))` streamed with `.values_list("room_id", ...)`
   over the same columns the used-hours contract needs, accumulating per `room_id`
   into a dict. Reuse the per-session contribution logic from `room_utilization`
   rather than restating it -- factor it into a small module-level helper taking
   the tuple of status and four datetimes and returning `(booked_seconds,
   used_seconds, is_in_flight)`, and have `room_utilization`, `room_heat_grid` and
   this function all call it. One definition of "used" is the whole point of the
   phase; three copies will drift.

4. Build one `RoomLoad` per room from the universe list, looking the totals up in
   the dict and defaulting to zeros on a miss.

5. Sort by utilization ascending, then by room code ascending for a stable
   tiebreak, so the head of the list is the least-used-rooms answer directly and
   two identical requests always produce the same order. Document that the caller
   takes a slice for a top-N view rather than the aggregate imposing a limit --
   the export in plan 07 needs the full set, and an aggregate that truncates
   cannot serve both.
