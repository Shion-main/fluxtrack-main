Binding, and every clause has drawn blood in this project before.

- IFO surfaces extend `templates/ifo/_console.html` and use the Franken `uk-*`
  vocabulary. The navy `.ft-*` vocabulary is Faculty and Guard only and must not
  appear on this page.
- Tables are `.tbl` inside `.table-wrap`, on a `uk-card uk-card-body p-0`, with
  `tbl__num` on numeric cells and `tbl__id` on identifier cells, paged through
  `web/pagination.py` and `{% include "_pager.html" %}`.
- No border-left accent stripes.
- Colour never carries meaning alone. Every heat cell pairs its shade with a
  visible number, following the `pill` precedent at `_cards.html:66` and the free-cell
  precedent at `room_detail.html:89` which pairs a dash with an `sr-only` label.
- The heat scale must meet WCAG AA contrast for its cell text at every step.
- Console nav hrefs are hardcoded literal paths, not `{% url %}` -- follow the
  local convention in `_console.html`. `is-active` and `aria-current` are both
  driven by the resolved url name in the same condition.
- `web/urls.py` is a flat named-pattern list with no namespace and no trailing
  slash, grouped by area.
- After ANY CSS edit, run `collectstatic`. A missed collectstatic has shipped an
  unstyled surface on this project twice.
- Restart the dev server before judging a template change; with `--noreload`
  Django serves a cached template and a correct edit looks like a failure.
</ui_contract>

  <action>
Create `templates/ifo/_heatgrid.html` as an includable partial rendering one
`HeatRow` per table row.

Mirror the printed timetable's skeleton from `room_detail.html:57-67` so the two
grids read as the same object: a `.tt` table in a `.table-wrap`, a
`<th scope="col" class="tt__time">` heading the time column, day headers walked
from the day axis, and each row's time in a `<th scope="row" class="tt__time">`
formatted the same way. This shared skeleton is the whole point -- an IFO admin
reading the dashboard grid and the printed room timetable must never have to
reconcile two different grids.

Each cell renders the block-and-day utilization percentage as visible text,
alongside the used hours. Colour is an addition to the number, never a substitute:
a reader who cannot distinguish the shades must still get the full answer from the
cell. A cell with a zero denominator -- a weekday the campus does not teach --
renders in the established free-cell idiom from `room_detail.html:89`, a dash
marked `aria-hidden` paired with an `sr-only` word, rather than a misleading 0%.

Give the table a caption or an adjacent legend that states the unit and explains
the scale in words, so the grid is self-describing when printed or screenshotted
into a report.

Extend `static/css/timetable.css` with the intensity steps rather than creating a
second stylesheet. One file must govern both grids or they will visually diverge
the first time either is touched. Add the scale as modifiers in the existing
vocabulary, alongside the current binary `.tt__on`, building the ramp over the
existing `--navy-tint` family so the heat grid stays inside the product's palette
instead of importing a new one.

Choose a five-step ramp and verify each step's text contrast against its
background meets WCAG AA for normal-size text. Where a dark step would fail

lightening the background -- the top of the ramp must remain visibly the top.

the next person to adjust a colour knows what constraint they are working inside.

heat ramp has the same problem and worse, since five shades of tint print as two.

keeping the numbers, which are the information.
