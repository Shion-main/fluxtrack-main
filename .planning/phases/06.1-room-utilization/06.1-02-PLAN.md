  <action>
Add `make_room_utilization_fixture(prefix="rutil")` beside `make_reporting_fixture`
in `scheduling/test_support.py`. Build it as a NEW independent fixture rather than
by mutating the existing one, so the documented faculty totals stay untouched.
Reuse the same idioms verbatim: the module's `_aware` helper for timezone-aware
datetimes, a `counter`/`_next()` closure for uniqueness, prefix-namespacing on
every unique column (`username`, `Building.code`, `Room.code`, `qr_token`,
`manual_code`, `Department.code`), and a `SimpleNamespace` return.

Seed exactly one `AcademicTerm` with `is_active=True` -- the ladder derivation
under test depends on there being exactly one, and a second active term is the
`make_session` trap in a different costume.

Shape it for rooms:

- Two Buildings, each with two Floors, so building and floor rollups have
  something to separate. Give each floor at least two physical rooms.
- One V-prefixed virtual room, in its own building, holding at least one ACTIVE
  online session with real actual times. Its code must start with an uppercase V
  and be prefix-namespaced in a way that keeps the V leading -- write it as the V
  first, then the prefix, so the exclusion still matches.
- A block ladder of at least three distinct start times with at least two
  distinct durations, so `campus_block_ladder` has something non-degenerate to
  derive and the per-block duration rule is exercised.
- Schedules on at least three weekdays including Saturday (`DayOfWeek.SAT`, value
  5) and none on Sunday, so `teaching_weekdays` demonstrably picks Saturday up
  and drops Sunday from the data rather than from a constant.

Seed one session of every used-hours shape, each on a documented date inside a
known week, and name each one on the returned namespace:

- `s_full` -- COMPLETED, `actual_start == scheduled_start`,
  `actual_end == scheduled_end`. Used equals booked.
- `s_absent` -- ABSENT, both actual timestamps NULL. Booked counts, used is zero.
- `s_early` -- COMPLETED, `actual_end` 30 minutes before `scheduled_end`. Also set
  `ended_early=True` and a `room_released_at` so the flag has a representative row
  even though the arithmetic does not key off it.
- `s_late` -- COMPLETED, `actual_start` 20 minutes after `scheduled_start`, normal
  end. Proves a late start also produces reclaimable hours.
- `s_overrun` -- COMPLETED, `actual_start` 5 minutes BEFORE `scheduled_start` and
  `actual_end` 5 minutes after `scheduled_end`. Proves the clamp holds and used
  can never exceed booked.
- `s_inflight` -- ACTIVE, `actual_start` set, `actual_end` NULL. The D-09 case.
- `s_noshow_stamped` -- COMPLETED with `actual_start` set and `actual_end` NULL.
  The anomalous shape: booked counts, used is zero.
- `s_virtual` -- ACTIVE in the V-prefixed room with full actual times. Must be
  invisible to every physical aggregate.
- `s_archived` -- a session whose Schedule has `status=ScheduleStatus.ARCHIVED`,
  with full actual times. Must be excluded by `_scoped_sessions` and so must not
  reach any room aggregate.

Write a docstring in the same voice as `make_reporting_fixture`'s, listing each
session by name with its shape and its intended contribution, and stating the
fixture's own documented totals: booked hours, used hours, wasted hours and the
distinct-block count. Compute those totals by hand and write them into the
docstring -- they are the contract the tests assert and the next reader's only
way to know what the fixture is supposed to mean.

Provide an `add_session(...)` escape hatch mirroring the existing fixture's, so
boundary tests can seed one more row without touching the documented totals.

- ABSENT adds its scheduled span to booked and nothing to used.
- The early-end session contributes its actual span, and the remainder appears in
  wasted hours.
- The late-start session likewise produces wasted hours.
- The overrun session's used contribution equals its booked span, not more --
  assert `used_hours <= booked_hours` for the whole fixture as well.
- The in-flight session is in neither booked nor used, and `in_flight` is 1.
- The COMPLETED-with-null-end session contributes booked hours and zero used.

`absent_hours + unused_held_hours` and that this figure is greater than zero for

regression back to keying off the `ended_early` flag.

<task type="auto">
  <name>Task 3: A human-runnable sanity check against seed_term</name>
  <files>scheduling/tests_reporting_rooms.py</files>
  <action>
This task produces no assertions against seeded data -- `seed_term` is randomized
per run and must never be a test dependency. It produces the recipe a human uses
to decide whether the number is believable.

Add a module-level docstring section (or a short block comment at the foot of
`scheduling/tests_reporting_rooms.py`) titled as the seed sanity check, giving a
copy-pasteable `manage.py shell -c` one-liner that:
- resolves the active `AcademicTerm`,
- computes the current reporting week via `web.reporting_common.reporting_range`
  with an empty request stand-in, or simply a hardcoded recent Monday-to-Sunday,
- calls `room_utilization` for that window,
- prints physical room count, blocks per day, teaching days in range, available /
  booked / used / wasted hours, both percentages, and the in-flight count.

Alongside it, record the four checks that make the output falsifiable, so a
reader knows what "believable" means rather than guessing:

1. `used_hours <= booked_hours <= available_hours`. A violation of the right-hand
   inequality means either double-booked rooms, sessions outside the derived
   ladder, or a denominator that lost its room-count factor.
2. `utilization_pct` should land well below saturation. `seed_term` runs a 60/20/20
   modality split and moves every online class into a virtual room, so a large
   share of physical capacity is genuinely idle; a figure near 100% means virtual
   rooms or virtual sessions leaked into the calculation.
3. `wasted_hours` must be clearly non-zero. `seed_term` marks roughly 8% of past
   sessions ABSENT (`seed_term.py:358-360`) and shaves 5 or 12 minutes off the end
   of roughly 40% of held ones (`seed_term.py:370-371`). A zero here means the
   waste metric is keyed off the `ended_early` flag, which the seeded data never
   sets.
4. `in_flight` should be non-zero when the command was run recently, because
   `seed_term` deliberately leaves `actual_end` NULL on the sessions running at
   seed time (`seed_term.py:384-395`). A zero here at a time of day when classes
   are in session means the D-09 branch is not being reached.

Note in the same block that these are observations to reason about, not
assertions to encode -- the dataset is regenerated with a different seed and any
test asserting a specific seeded figure will be flaky.
