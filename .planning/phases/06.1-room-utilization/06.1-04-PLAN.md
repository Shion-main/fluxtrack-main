<traps>
- **`room_timetable` cannot be reused for this.** It is per-room and
  Schedule-based, so its cells are BOOKINGS -- and D-03 is explicit that a booking
  count "is exactly the mistake this phase exists to correct". Reuse the ladder
  (Plan 01 already extracted it) and the half-open occupancy rule; build the
  aggregation new, over `Session`.
- **The half-open rule is load-bearing.** A session occupies every block where
  `block.start <= slot < session_end_time`, mirroring `room_state.py:131`. That is
  what makes a double-length class fill two rows with no rowspan bookkeeping. A
  session assigned only to its starting block will understate every long class.
- **Weekday occurrences are not uniform across a range.** A Monday-to-Sunday week
  has exactly one of each weekday, but an arbitrary range does not. The per-cell
  denominator must count how many times THAT weekday actually falls in the range,
  not divide the total day count by seven.
- **MSSQL.** No unbounded `pk__in`; materialize with `list()` before any follow-up
  query inside an `.iterator()`.
</traps>

  <action>
Add to `scheduling/reporting.py`, after `room_utilization`.

Define a `HeatCell` dataclass: `day` (the `DayOfWeek` int), `used_hours`
(Decimal), `booked_hours` (Decimal), `available_hours` (Decimal),
`utilization_pct` (int), `session_count` (int), `room_count` (int, distinct
physical rooms used in that cell). Define a `HeatRow` dataclass: `block` (the
`CampusBlock`) and `cells` (list of `HeatCell` in `DayOfWeek.choices` order).

Define `room_heat_grid(*, start, end, term, as_of=None)`, keyword-only. Return
None when `campus_block_ladder(term)` is None, matching the ladder's own contract
and `room_timetable`'s -- the caller already has to handle a None ladder, so this
adds no new burden.

Implementation:

1. Derive the ladder and the teaching weekdays once. Build a map from each weekday
   to the number of times it occurs in the effective range (`start` through
   `min(end, as_of)` when `as_of` is set). Compute it by walking the dates once,
   not by dividing the span by seven -- an arbitrary range does not contain equal
   numbers of each weekday.

2. Count physical rooms once with `_physical_rooms().count()`.

3. Take `_scoped_sessions(start=start, end=end, as_of=as_of)`, pass through
   `_exclude_virtual`, and stream
   `.values_list("schedule__day_of_week", "schedule__start_time", "schedule__end_time", "room_id", "status", "scheduled_start", "scheduled_end", "actual_start", "actual_end")`
   with `.iterator()`. Take the day and the wall-clock span from the schedule so a
   session lands on the same grid the printed timetable draws; take the durations
   from the session's own timestamps so the value is real usage, not the booking.
   Do not query inside the loop.

4. For each row, determine the blocks it occupies with the half-open rule --
   every block whose `start` satisfies `block.start >= schedule_start_time` and
   `block.start < schedule_end_time`. Distribute the session's booked and used
   seconds across those blocks. Distribute proportionally by block duration rather
   than assigning the full span to each, so a two-block class contributes its
   hours once in total across the grid and the grid's totals reconcile with
   `room_utilization`. State this reconciliation property in the docstring -- it is
   the property that stops the dashboard card and the grid from disagreeing, and
   plan 06 renders both on the same page.

5. Track distinct room ids per cell in a set, and take its length for
   `room_count`. Sets of integer ids are bounded by the physical room count, so
   this is safe; do not turn them into a `pk__in` query.

6. Per-cell denominator: `physical_room_count * weekday_occurrences[day] * block.duration_seconds`.
   Compute `utilization_pct` through `_hours_pct`, which already absorbs a zero
   denominator.

7. Return the list of `HeatRow`. Give `room_heat_grid` a docstring naming T2 and
   citing D-03 for why the cells aggregate Sessions rather than Schedules, and
   noting explicitly that `room_timetable` was NOT reused because it is per-room
   and Schedule-based -- so a later reader does not "simplify" it into that
   function and silently reintroduce a booking count.
  </action>
