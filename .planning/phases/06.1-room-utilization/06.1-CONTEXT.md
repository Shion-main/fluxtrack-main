# Phase 06.1: Room Utilization & IFO-09 Closure - Context

**Gathered:** 2026-07-19
**Status:** Ready for planning

<domain>
## Phase Boundary

FluxTrack is "Faculty Attendance **and Facility Utilization**", but every report
built so far measures only whether faculty showed up. Nothing measures whether
the room was used. This phase builds the missing half.

**In scope (T1 + T2, per the owner's decision 2026-07-19):**

- **T1 — the number.** Room-hours booked vs room-hours used; utilization rate =
  used / available; **wasted room-hours** (absences + early releases) expressed
  as reclaimable capacity — the metric a facilities office can act on.
- **T2 — where and when.** Day x block heat grid reusing the existing room
  timetable ladder; per-building and per-floor table; least-used rooms; saturated
  blocks.
- **IFO-09 closure.** The SRS names four dashboard cards — Faculty, **Room
  Occupancy in session-hours**, Sessions, Absences. `templates/ifo/_cards.html`
  renders Faculty, Sessions, Absences and **Attendance %**: the room-occupancy
  card was swapped for a second attendance metric. Restore it.

**Out of scope:**
- **T3 right-sizing** (seat utilization enrolled/capacity; physical room-hours
  freed by the online+blended share) — deliberately deferred. `enrolled_count`
  trustworthiness across the imported term is unproven and would need its own
  verification before any seat metric is credible.
- The scan resolver, `scheduling/resolver.py`, and the shared no-show predicate.
- Anything Entra/AWS (Phase 8).
</domain>

<decisions>
## Implementation Decisions

### D-01 — The denominator is the campus block ladder (LOCKED by the owner)
`available_hours = physical rooms x 11 campus blocks x teaching days`, excluding
virtual rooms (`Room.is_virtual`, the V-prefix convention).

Rejected alternative: a fixed 7am-9pm clock over every calendar day. It counts
hours the campus never timetables, so utilization looks far worse than it is and
the "wasted hours" figure stops being actionable — a 7am block nobody schedules
is not waste, it is simply not a teaching slot. The ladder is also already the
grid the room timetable renders, so the dashboard and the printed timetable
cannot disagree about what a slot is.

### D-02 — No schema change; every field already exists
`scheduled_start`/`scheduled_end` (booked hours), `actual_start`/`actual_end`
(used hours), `status` (ABSENT = booked-but-empty), `ended_early` +
`room_released_at` (reclaimed time), `Room.capacity` + `Schedule.enrolled_count`
(T3, deferred), `Room.is_virtual` (exclude from physical capacity),
Building/Floor (rollups). If a plan reaches for a migration, that is a signal the
metric is being defined wrong — stop and re-read this decision.

### D-03 — "Used" means actual, not booked
A session that was booked but marked ABSENT contributes to BOOKED hours and zero
USED hours; that difference IS the waste metric. A session that ended early
contributes `actual_end - actual_start`, not the scheduled span, and the
remainder counts as reclaimed. Never infer usage from the schedule alone — the
existing room page's "N of 77 slots used" is a BOOKING count and is exactly the
mistake this phase exists to correct.

### D-04 — Virtual rooms are excluded from physical capacity
Online classes no longer occupy physical rooms (commit `08a9cce`). A V-prefixed
room must never appear in a physical utilization denominator or a heat grid cell,
or the rate is diluted by rooms that do not exist.

### D-05 — Reuse the Phase 6 aggregate conventions
`scheduling/reporting.py` is the established aggregate layer: DB-side
`Count(filter=Q)`, `safe_card` isolation so one raising card cannot take out the
page, filtering on local `Session.date` NOT UTC `scheduled_start`. Room
aggregates join that module and follow the same rules. `safe_card` is not
optional — a utilization card that raises must degrade to a message, not a 500.

### D-06 — CORRECTION to D-01: the ladder is DERIVED, and so are the days
D-01 said "11 campus blocks x teaching days". **The 11 is not a constant and must
not be hardcoded.** `web/room_state.py:110-113` derives the ladder as
`sorted(set(...))` of every distinct `Schedule.start_time` in the active term; 11
is merely what the current dataset contains. The denominator must use
`len(slots)` computed at query time.

Likewise **days are derived, not assumed.** `DayOfWeek` includes SAT and SUN
(`scheduling/models.py:12-19`), so the ladder's own definition implies 7 — but
MMCM schedules real Saturday classes and no Sunday ones. Deriving the teaching
days from the term's actual schedules (the same way blocks are derived) picks
Saturday up and drops Sunday automatically, from data rather than a hardcoded 5
or 7. This is the identical rationale D-01 was chosen for: a slot nobody
schedules is not waste, and neither is a day nobody schedules.

### D-10 — REFINES D-01/D-06: the denominator is a (day, block) CELL SUM
Available capacity is summed over the **(day, block) cells the term actually
timetables**, not over `days x blocks` as a cross-product:

    available = SUM over timetabled (day, block) cells of
                (physical_rooms x block_duration)

**Why this replaces the cross-product.** Wave 1 shipped `days x blocks` and it
exposed the flaw immediately: the seeded term contains exactly ONE Sunday class
(`AR200P-2 B199`, R102, 15:45-20:45) against 289 on Saturday. Under the
cross-product that single row promoted Sunday to a full teaching day and added
125 rooms x 11 blocks x 13.75h = 1,718h to a 12,031h denominator, moving campus
utilization from ~27% to 23%. One outlier row must not move the headline metric
by four points.

This is the same rationale D-01 was chosen for, applied one level finer: a BLOCK
nobody schedules is not waste, and neither is a (day, block) cell nobody
schedules. Sunday now contributes only its one real block instead of all eleven.
Still fully derived — no thresholds, no magic numbers, no hardcoded day list, so
D-06 is preserved rather than weakened.

Ladder-tiling is confirmed sound and does NOT need revisiting: the derived ladder
is 11 blocks, 07:00-19:30, each exactly 75 minutes, spaced exactly 75 minutes
apart, so `block.start + duration == next.start` for every adjacent pair and
summed durations do not overstate wall-clock capacity.

### D-07 — Five KPI cards, not four (LOCKED by the owner)
Final row: **Faculty | Room Occupancy | Sessions | Absences | Attendance %.**
The SRS names four and assigns the fourth slot to Room Occupancy, but Attendance
% is genuinely useful and stakeholders may already read it, so it is kept rather
than displaced. The grid is currently `lg:grid-cols-4`
(`templates/ifo/_cards.html:11`) and needs a layout adjustment to carry five.

### D-08 — Physical-only is expressed as `exclude(code__startswith="V")`
`Room.is_virtual` is a PROPERTY (`campus/models.py:44-54`), not a field, so it
cannot appear in a `.filter()`. The in-production idiom is
`exclude(code__startswith="V")` (`seed_term.py:253-255`). `Room.code` uses the
DB-wide CI collation (only `qr_token`/`manual_code` are CS_AS), so this already
matches a lowercase `v`.

### D-09 — Used-hours needs an explicit null policy
`actual_start`/`actual_end` are nullable, and `seed_term` genuinely leaves
`actual_end` NULL on in-flight ACTIVE sessions. The plan must state what an
in-flight session contributes to used-hours (e.g. clamp to `now`, or exclude)
rather than letting a NULL silently become zero. Also note `Session.date` is
LOCAL while the timestamps are UTC — filter on the former, compute on the latter.

### Claude's Discretion
- Exact function names and signatures in the aggregate layer.
- Heat-grid colour scale (must be WCAG-AA and never colour-alone — pair every
  cell with a value or pattern, per the established rule).
- Whether utilization gets its own IFO console page or extends the dashboard.
</decisions>

<specifics>
## Specific Ideas

- The owner's framing: "FluxTrack is Faculty Attendance **and Facility
  Utilization**" — every report so far measures only the first half.
- Wasted room-hours should read as reclaimable capacity, not as a scolding
  statistic: the actionable sentence is "N room-hours were booked and unused this
  week", which a facilities office can convert into freed bookable slots.
- The heat grid should sit on the same block ladder as the printed timetable so
  an IFO admin reading both never has to reconcile two different grids.
