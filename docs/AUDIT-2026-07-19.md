# FluxTrack Full-Codebase Gap Audit — 2026-07-19

Five parallel deep audits (requirements coverage, security/access control, domain
correctness, frontend/UX/PWA, ops/testing readiness) run against the codebase as of
`e2435b4` (Phase 07 complete, Phase 8 not started). Every finding below was verified
in code — docs were treated as claims to check, not as truth. Test suite at audit
time: **930 tests, 3 failures** (the known pre-existing dev-login/home-redirect
asserts — diagnosed in §6.3; none in the attendance state machine), 2 skips.

**Verdict in one line:** the feature surface is genuinely ~95% built and the roadmap's
completion claims held up under code verification. The real remaining gaps are
(1) a handful of correctness bugs that can produce *wrong attendance records* — the
one thing the product promises not to do, (2) a set of production-infrastructure
gaps that Phase 8 as currently scoped will miss, (3) pre-cutover security hardening,
(4) SRS/traceability drift that a capstone panel will notice, and (5) floor-UX trust
issues.

---

## 1. Wrong-record bugs (domain correctness) — fix before anything else

These break the core promise ("the attendance record is trustworthy") with real,
traced scenarios. Ordered by payback.

### HIGH

**H1 — FAC-08 online self-start doesn't propagate merged-present** · `web/faculty.py:689-708`
`online_start` writes only the anchor session; `propagate_merged_present` is never
called (the checker path and both F2F scan paths do call it). An instructor with
co-scheduled online sections (~27% of instructors have co-scheduled slots) starts
one section; the sibling stays SCHEDULED and the sweep marks it **Absent**. Recorded
as a Phase-07 scope call, but it is the largest systematic source of false Absents,
and the Home card visually teaches "one action covers the group" while the online
surface doesn't. *Fix: one line inside the existing `transaction.atomic()` —
`propagate_merged_present(session, session.actual_start, request.user)` — plus tests.*

**H2 — Online Flag-not-present absents siblings with no grace check** · `web/checker.py:300-307`, `scheduling/merge.py:153-183`
`propagate_merged_absent` flips every SCHEDULED sibling ABSENT immediately —
bypassing `is_no_show_past_grace`, which every other ABSENT path requires. ABSENT is
terminal, so a checker who flags one sibling of a merged group 3 minutes after start
permanently blocks a within-grace faculty start (and the other checker's verify) on
the rest of the group. The F2F/mixed merge arm (`course_code` match) means an online
flag can even absent a same-course F2F sibling. *Fix: gate sibling absenting on the
grace predicate (flag the anchor now, let the sweep absent siblings), or make
flag-driven ABSENT reversible by a within-grace start/verify.*

**H3 — Offline replay applies a queued verification to whatever session occupies the room NOW** · `web/checker.py:491-493`
Replay re-derives the room's session at *replay time* and applies the stored action
to it, preserving the original `scanned_at`. A scan verified offline at 09:20 that
replays at 11:05 verifies the 11:00 class the checker never saw — and leaves the
09:00 class unverified. *Fix: queue the session_id client-side (or match `scanned_at`
to a session window) and write `replay_conflict` when the then-session ≠ the
now-session.*

### MEDIUM

**M1 — All idempotency/rate-limit/replay-dedupe guards sit on per-process LocMemCache** · no `CACHES` in `config/settings.py`
`scan-idem` (`web/scan.py:177-180`, also non-atomic get-then-set — the old idiom
checker.py already fixed), `checker-idem` (`web/checker.py:415-418,582-583`), and the
CHK-08 replay dedupe `checker-replay:{uuid}` with `timeout=None`
(`web/checker.py:461-519`). LocMem is per-worker, lost on restart, and culls at 300
entries — so the replay dedupe *forgets*: a restart between two replays re-applies
every queued item; a long replay batch can evict its own keys; any multi-worker
deploy voids all of it. `client_uuid` is never persisted to DB. *Fix: shared cache
backend (Redis or `createcachetable` DB cache) for idem/rate-limit keys, and persist
replay `client_uuid` with a unique constraint — cache-only idempotency for an offline
attendance system is structurally wrong.*

**M2 — `confirm()` re-applies without re-gating or idempotency; token replayable for 180 s** · `web/scan.py:186-203`
No resolver re-run, no status guard on ROOM_OCCUPIED / EARLY_END / WRONG_ROOM, no
idem key. Double-POST of a ROOM_OCCUPIED confirm overwrites `actual_start` with the
later instant (lateness evidence destroyed); a confirm after the sweep marked ABSENT
writes ACTIVE anyway; WRONG_ROOM confirm can move a COMPLETED session or collide two
ACTIVE sessions. *Fix: re-fetch and refuse unless status matches the outcome's
precondition; single-use nonce in the token.*

**M3 — Ad-hoc bookings beyond the materialize horizon silently collide with timetabled classes** · `web/ifo.py:857`, `ops/availability.py:113-139`, `scheduling/.../materialize_sessions.py:143-147`
`room_is_free` sees Sessions/Bookings/reservations but not `Schedule` rows; sessions
exist only ~7–14 days out; the materializer never checks Bookings. A booking 3 weeks
out is accepted, then the weekly class materializes into the same room+window, and
nothing flags it (JOB-02c only counts ACTIVE Sessions). Distinct from the recorded
T-07-27 accept. *Fix: booking_create also tests Schedule weekday/window occurrences,
or materialize refuses/notifies on an active Booking collision.*

**M4 — Scan occupancy ignores `room_released_at`** · `web/scan.py:159-160`
A manually released (IFO-08) but still-ACTIVE ghost session still triggers
ROOM_OCCUPIED → the next faculty's confirm force-COMPLETES it with `actual_end=now`,
hours late — inflating used-hours in utilization and recording a phantom handover.
*Fix: add `.filter(room_released_at__isnull=True)` to the occupying query.*

**M5 — Resolver candidate selection ignores the scanned room; back-to-back sessions hijack check-ins** · `scheduling/resolver.py:82-87`
First window-containing SCHEDULED session wins, room compared only afterwards. A
never-checked-in earlier session captures a scan meant for the next session in a
different room; confirming the WRONG_ROOM dialog rewrites the *earlier* session's
room. *Fix: prefer a window-containing candidate whose room matches the scanned room.*

**M6 — Withdraw/approve/reject races: PENDING re-gate has no row lock** · `scheduling/services.py:319-331,344-368,504-511`
Plain `.get()` + status check inside the transaction; under READ COMMITTED a
withdraw and an approve can both read PENDING — approve applies full room
consequences over a request whose stored status may end "Withdrawn". *Fix:
`select_for_update()` in all three transitions (mssql-django supports it).*

**M7 — Sweep saves from a stale instance** · `scheduling/jobs.py:77-89`
Candidates are materialized then `save(update_fields=["status"])` — a check-in that
commits in the sliver between materialize and save is overwritten with ABSENT,
leaving `actual_start` + a `session.checked_in` audit row contradicting the status.
*Fix: status-guarded `filter(pk, status=SCHEDULED).update(...)`, count only rowcount==1.*

### LOW / verified-sound (for the record)

- Grace boundary consistent everywhere — single predicate, never copied (L1). ✓
- Timezone discipline holds; midnight-crossing unreachable with real data (L2). ✓
- Checker online verify has no grace gate (deliberate-looking; worth a recorded
  decision) (L3).
- Flag on an ACTIVE-verified group splits state (anchor ABSENT, siblings ACTIVE) —
  guard correctly protects independently-verified siblings (L4).
- R116 double-booking behaves as designed; handover truncation invisible in
  early-end stats; earlier ABSENT session can shadow a later SCHEDULED one from
  checker action on overlapping windows (L5).
- Notifications sound: single write path, mutes suppress list+push, push
  fault-isolated; pushes older than the 15-min outbox window after downtime are
  never sent (bell still shows them) (L6).
- Aggregates match the state machine (MERGED = held-but-unverified as designed);
  archived-schedule absences vanish from history (documented A5) (L7).
- Born-released materialize hook not atomic with session creation; no DB unique on
  `(schedule, date)` (L8 — plausible, crash-window only).

## 2. Security & access control

**The good news first:** role gating is essentially complete — every data-bearing
view carries the right `*_required` decorator; Guard read-only is enforced with
405s; IDOR discipline is real (ownership re-checked inside service transactions,
Dean scoping via `get_object_or_404(..., department=...)`, checker re-gates
server-side, import commit token from session not form); CSV exports all route
through the single `csv_safe`; uploads validate-then-re-encode with EXIF strip; no
secrets in git history; Teams-link host allowlist defeats lookalike domains;
deactivated users blocked on all three auth paths; xlsx parsing byte- and row-capped.

**HIGH — `seed_demo` creates `admin`/`devpass123` superuser with no DEBUG guard** · `accounts/management/commands/seed_demo.py:73-80,103`
`seed_term` refuses under `DEBUG=False` without `--force`; `seed_demo` has no guard.
`ModelBackend` stays in `AUTHENTICATION_BACKENDS` in prod (break-glass) and `/admin/`
is always mounted with password auth — one accidental `seed_demo` against a
prod/staging DB is a working known-password superuser with all seven roles. *Fix:
same DEBUG guard as seed_term; random/forced-change passwords for superusers.*

**MEDIUM — fail-open production defaults** · `config/settings.py:36-38` and absent settings
`SECRET_KEY` falls back to `"dev-insecure-change-me"` (forgeable sessions, CSRF, and
the signed scan-confirm token), `DEBUG` defaults **True**, `ALLOWED_HOSTS` defaults
`"*"`. Zero of: `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE`, `SECURE_SSL_REDIRECT`,
HSTS, `SECURE_PROXY_SSL_HEADER`. *Fix: `not DEBUG`-keyed secure block; refuse to boot
on placeholder SECRET_KEY when DEBUG=False; default DEBUG False, hosts empty.*

**LOW** — `logout_view` GET-reachable (logout CSRF) `web/views.py:95-97`;
notification mark-read on GET `web/notifications.py:45-63`; `ifo.live` undecorated
(redirect-only, no leak) `web/ifo.py:190-193`; scan idempotency non-atomic
get-then-set (align to `cache.add`) `web/scan.py:177-180`.

## 3. What Phase 8 as scoped will MISS (ops/deploy readiness)

Phase 8 says "SSO cutover + EC2/RDS + Tailwind build". These are the gaps inside
that framing:

1. **Multi-worker correctness (CRITICAL)** — everything in M1 above, plus both
   manual-code rate limiters (`web/scan.py:50-55`, `web/checker.py:167-172,210-213`)
   are per-worker: effective limit = N × policy, reset on worker recycle. Either a
   shared cache backend lands in Phase 8, or `--workers 1` is a documented decision.
2. **HTTPS/proxy config** — with no `CSRF_TRUSTED_ORIGINS`/`SECURE_PROXY_SSL_HEADER`,
   Django behind Nginx TLS will **403 every htmx POST on day one**.
3. **Hardcoded SSO redirect URI** — `settings.py:155-157` pins
   `http://localhost:8000/...`; prod SSO fails AADSTS50011 until env-driven. (And the
   Entra app registration currently sits in a sandbox tenant, not MMCM's — the live
   UAT from 03.1-05 is still blocked on redirect-URI registration.)
4. **Media serving** — `/media/` served only under DEBUG (`config/urls.py:16-17`);
   WhiteNoise covers static only → **profile photos 404 in prod**. The naive Nginx
   `location /media/` fix would expose weekly-report CSVs/PDFs and staged imports
   that are currently auth-gated through streaming views — media needs a deliberate
   public/private split.
5. **Scheduler resilience** — `ops/jobrun.py:34` creates the JobRun row outside the
   try and deliberately omits `close_old_connections`: one dropped RDS connection
   makes every tick of every job raise forever, silently (the failure-alert path
   lives inside the dying process; SYS-04 is display-only with no staleness alarm).
   Weekly report is the one non-self-healing job (Mon-06:00 cron on MemoryJobStore —
   downtime over Monday morning loses that week; `generate_week_reports` is already
   idempotent, so a startup backfill is a cheap fix). Push outbox only scans a
   15-min window.
6. **Data lifecycle vs the 10 GB RDS cap** — JobRun grows ~2.1 M rows/yr from the
   15-second push job alone, with **no pruning job anywhere**; AuditLog, Notification
   ("keep-all" explicit), and `django_session` (`clearsessions` never scheduled) are
   append-forever. **No backup story at all**, and media (photos, weekly-report
   archive) is EC2-local disk outside RDS snapshots. Term rollover is undesigned:
   `reset_term` deletes the attendance history HR exports from and PROTECT-skips
   schedules referenced by modality items; `DEFAULT_TERM` hardcoded.
7. **Frontend vendoring** — Phase 8 scopes only the Tailwind build, but htmx,
   Franken JS, and html5-qrcode (unmaintained since 2022, carries the core scan
   gesture, no SRI) are also CDN-only, and the service worker caches same-origin
   only — so the PWA offline story is cosmetic until all four are self-hosted.
8. **Dependency hygiene** — gunicorn absent from requirements.txt; no lock file
   (broad ranges resolve differently at deploy time); DRF installed but unused;
   `pypandoc_binary` belongs in requirements-dev; msodbcsql18/unixODBC provisioning
   and VAPID `keys/` runbook undocumented; real-term xlsx paths are gitignored and
   must ship to EC2 out-of-band.
9. **Misc** — no `CONN_MAX_AGE`; no LOGGING config (one logger in the codebase;
   prod 500s go to Gunicorn stderr and nowhere else); no health endpoint for
   Nginx/systemd checks.

   spec/code contradiction in the project.
2. **AUTH-02 / REST API**: SRS describes a backend-issued JWT and a DRF/JSON API;
   reality is Django sessions + server-rendered htmx, no DRF surface. The
   session-vs-JWT decision was flagged in USE_CASES.md as "decide explicitly" and
   was then shipped implicitly.
3. **S3**: SRS states photos/reports in S3; reality is FileSystemStorage
   (`ops/models.py:251` docstring is aspirational).
4. **JOB-02**: SRS still requires timer-based room release + `room_hold_minutes`;
   deliberately cut 2026-07-03 (release is MOD-03/born-released/IFO-08 only). SRS
   §4.13 and the §8 policy row were never amended.
5. **CHK-02**: the online Teams-link verification path (Verify activates the
   session) exists nowhere in the SRS — v1.2 amended CHK-03 but not CHK-02; FAC-08
   coexistence never documented.

### Offline / PWA

  `base.html:8`).
